# 代码结构规范

## ⛔ 绝对禁止（AI必须严格遵守）

**以下操作绝对禁止，违反即为错误：**

1. ❌ **禁止修改用户指定变量区域的任何内容**（包括变量名、变量值、注释）
2. ❌ **禁止添加用户未指定的变量**（用户指定几个就是几个）
3. ❌ **禁止删除已实现的功能**（规范服务于功能，不是牺牲功能）
4. ❌ **禁止在调试逻辑中添加包装函数、页面配置、硬编码参数**
5. ❌ **禁止未经测试就提交修改**

**迁移旧版代码时：**
- 用户指定变量必须与旧版**完全一致**（数量、名称、值、注释）
- 功能必须与旧版**完全一致**

## 模块结构（6部分）

新建模块必须包含以下6个部分，顺序不得更改：

### 1. 文件头
```python
# -*- coding: utf-8 -*-
```

### 2. 文档字符串
```python
"""
模块名称：XXX 
设计思路: YYY 
模块隔离: ZZZ
"""
```

### 3. 导入部分
- 标准库导入
- 第三方库导入
- 项目内部模块导入
- 不导入未使用的模块

### 4. 用户指定变量区域
```python
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
USER_XXX = 16  # 中文说明
# *********************************
```

无用户指定变量时：
```python
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
# （用户未指定变量）
# *********************************
```

### 5. 类/函数定义
- 核心功能实现
- 明确输入输出类型
- 函数长度≤20行

### 6. 调试逻辑
```python
# *** 调试逻辑 ***
if __name__ == "__main__":
    ft.run(lambda page: page.add(XXX.create()))
```

## 核心原则

### 功能保护（最高优先级）
- **功能不可丢失**：修改代码时必须保持原有功能不变
- **规范服务于功能**：规范是为了更好地实现功能，而不是牺牲功能
- **冲突处理**：如果规范与功能冲突，应寻找其他实现方式，而不是删除功能
- **验证要求**：修改代码后必须验证原有功能仍然正常工作
- **禁止操作**：
  - ❌ 禁止为了符合规范而删除已实现的功能
  - ❌ 禁止简化代码导致功能丢失
  - ❌ 禁止未经测试就提交修改

### 用户变量保护
- **区域必须存在**：每个模块必须包含用户指定变量区域
- **禁止修改**：AI不得修改用户指定变量区域的任何内容
- **变量值必须生效**：用户指定的变量值必须在代码中实际使用
- **命名规范**：必须使用`USER_`前缀，每个变量必须添加中文注释
- **优先级**：函数参数 > 用户指定变量 > 配置文件 > 代码默认值

### 模块隔离
- **禁止往前覆盖**：不修改被调用方的用户指定变量
- **通过参数传递**：调用方通过函数参数传递配置
- **公开接口访问**：只通过公开接口访问模块

### 调试逻辑规范
- **标准格式**：`ft.run(lambda page: page.add(XXX.create()))`
- **注释标记**：必须有`# *** 调试逻辑 ***`注释
- **只能替换模块名**：`XXX`替换为被测模块名
- **禁止添加**：
  - ❌ 禁止包装函数：`def main(page):`
  - ❌ 禁止页面配置：`page.title`, `page.padding`
  - ❌ 禁止硬编码具体值：传入具体值（如`title="测试"`）
  - ❌ 禁止初始化代码
- **允许引用用户指定变量**：`ft.run(lambda page: page.add(Divider.create(width=USER_CONTAINER_WIDTH)))`
- **窗口设置处理**：入口模块应通过 `on_mount` 事件自动设置窗口属性，而不是在调试逻辑中设置

### 测试默认数据
- **默认值处理**：create函数必须为所有参数提供默认值，确保测试代码无需额外配置即可运行
- **测试默认数据**：默认值应提供必要的测试数据，确保组件在测试时能够展示真实功能
  - 示例：下拉框应提供默认选项列表，输入框应提供默认提示文本，卡片应提供默认标题和内容
  - 目的：测试代码能够真实测试组件的运行情况，而不是创建空组件
- **可见性保证**：组件设计必须确保在默认参数下可见，避免因布局或尺寸问题导致测试时不可见

### 代码质量
- **函数长度**：≤20行
- **类拆分**：超过300行考虑拆分
- **无冗余**：移除未使用的变量、导入和重复代码
- **禁止别名**：禁止在模块中创建或导出中文别名

## 关键规范

| 规范项 | 要求 |
|--------|------|
| 文档字符串 | 单行格式：`"""模块名称：XXX \| 设计思路 \| 模块隔离原则"""` |
| 用户变量区域 | 必须存在，用户未指定时标注 `# （用户未指定变量）` |
| 用户变量命名 | 必须使用`USER_`前缀，每个变量必须添加中文注释 |
| 用户变量生效 | 用户指定的变量值必须在代码中实际使用 |
| 导入顺序 | 标准库 → 第三方库 → 项目内部模块 |
| 函数参数 | 所有参数应有默认值，`=`两边不加空格 |
| 默认值传递 | 从被调用模块导入默认值，禁止硬编码 |
| 调试逻辑注释 | 必须有`# *** 调试逻辑 ***`注释 |
| 调试逻辑格式 | 标准格式，只能替换模块名，禁止添加其他代码 |
| 错误处理 | 使用 try-except 捕获异常，避免裸 except，记录错误信息 |
| 命名规范 | 类名使用 PascalCase，函数和变量使用 snake_case，常量使用全大写 |
| 注释规范 | 关键逻辑添加注释，注释应简洁明了，避免冗余 |
| 性能优化 | 避免不必要的计算，合理使用缓存，优化渲染性能 |
| 安全规范 | 避免硬编码敏感信息，使用配置文件或环境变量 |
| 代码风格 | 统一缩进（4个空格），行宽不超过80字符，保持代码整洁 |

## AI检查清单

创建或修改模块时，AI必须检查以下项目：

**功能保护检查（最高优先级）**：
- [ ] 是否保持了所有原有功能？
- [ ] 是否验证了修改后功能仍然正常工作？
- [ ] 是否避免了为了符合规范而删除功能？

**代码结构检查**：
- [ ] 文件头是否正确？
- [ ] 文档字符串是否包含模块名称、设计思路、模块隔离？
- [ ] 导入顺序是否正确（标准库→第三方库→项目内部模块）？
- [ ] 用户指定变量区域是否存在？
- [ ] 用户指定变量是否使用`USER_`前缀？
- [ ] 用户指定变量是否添加了中文注释？
- [ ] 用户指定变量值是否在代码中实际使用？
- [ ] 调试逻辑是否有`# *** 调试逻辑 ***`注释？
- [ ] 调试逻辑是否使用标准格式？
- [ ] 调试逻辑是否只替换了模块名？
- [ ] 调试逻辑是否未传入任何硬编码参数？
- [ ] 调试逻辑是否未添加初始化代码？
- [ ] 函数长度是否≤20行？
- [ ] 是否移除了未使用的变量和导入？
