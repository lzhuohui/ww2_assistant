# AI执行标准 - 核心原则

## 一、核心原则

### 1.1 模块完整性

新建模块必须包含完整的5个部分，顺序不得更改：

1. **文件头**：编码声明和文档字符串
2. **导入**：必要的模块导入
3. **用户指定变量**：集中管理，AI不得修改
4. **类/函数定义**：核心功能实现
5. **调试逻辑**：标准化测试入口

### 1.2 用户变量保护

#### 核心原则
- **区域必须存在**：每个模块必须包含用户指定变量区域，不可省略
- **禁止修改**：AI不得修改、删除或添加用户指定变量区域的任何内容
- **禁止覆盖**：不得在同一模块内二次覆盖用户指定变量
- **变量值必须生效**：用户指定的变量值必须在代码中实际使用
- **优先级**：函数参数（运行时） > 用户指定变量（模块级） > 配置文件（全局） > 代码默认值（静态）
- **隔离保护**：调用方不得直接修改被调用方的用户指定变量

#### 配置优先级规则

**优先级顺序：**
```
函数参数（运行时） > 用户指定变量（模块级） > 配置文件（全局） > 代码默认值（静态）
```

**详细说明：**

| 优先级 | 配置来源 | 说明 | 作用范围 | 示例 |
|-------|---------|------|---------|------|
| **1** | 函数参数（运行时） | 运行时传入的参数，具有最高优先级 | 当前函数调用 | `ContainerIcon.create(icon_size=30)` |
| **2** | 用户指定变量（模块级） | 模块级别的配置覆盖，仅在本模块内生效 | 本模块 | `USER_ICON_SIZE = 20` |
| **3** | 配置文件（全局） | 全局默认配置，适用于所有模块 | 全局 | `配置.获取尺寸("图标", "size_medium")` |
| **4** | 代码默认值（静态） | 代码中的硬编码默认值，作为最后兜底 | 代码 | `def create(icon_size: int = 20)` |

**配置传递规则：**
- **禁止往前覆盖**：调用方不能直接修改被调用方的用户指定变量
- **通过参数传递**：调用方通过函数参数传递配置，覆盖被调用方的默认行为
- **模块隔离**：每个模块的用户指定变量仅在本模块内生效

**正确示例：**
```python
# 被调用模块：容器图标.py
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
USER_ICON_SIZE = 20  # 仅在容器图标模块内生效
# *********************************

@staticmethod
def create(icon_size: int = USER_ICON_SIZE) -> ft.Container:
    # 优先级：icon_size参数 > USER_ICON_SIZE > 配置文件 > 代码默认值
    return ft.Container(...)

# 调用方模块：容器标题.py
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
USER_ICON_SIZE = 24  # 仅在容器标题模块内生效
# *********************************

@staticmethod
def create() -> ft.Container:
    # 通过函数参数传递配置，不覆盖被调用方的用户指定变量
    icon = ContainerIcon.create(icon_size=USER_ICON_SIZE)  # 传入 24
    
    # 或者使用被调用方的默认配置
    icon2 = ContainerIcon.create()  # 使用 USER_ICON_SIZE = 20
```

**错误示例：**
```python
# ❌ 错误：往前覆盖被调用方的用户指定变量
# 调用方模块
USER_ICON_SIZE = 24

@staticmethod
def create() -> ft.Container:
    # 直接覆盖被调用方的用户指定变量，违反模块隔离原则
    icon = ContainerIcon.create()  # 错误：期望覆盖为 24
```

#### 用户指定变量使用场景

**适用场景：**
1. **个性化定制**：用户需要对特定模块进行个性化配置
2. **临时调整**：用户需要临时调整某个参数进行测试
3. **特殊需求**：某个模块有特殊的使用场景，需要非标准配置
4. **快速原型**：用户需要快速创建原型，暂时跳过配置文件

**不适用场景：**
1. **全局配置**：应使用配置文件管理全局配置
2. **频繁变更**：频繁变更的配置应使用配置文件
3. **多模块共享**：多个模块共享的配置应使用配置文件

**使用原则：**
1. **优先使用配置文件**：默认配置应放在配置文件中
2. **仅在必要时使用**：仅在配置文件默认值不满足时使用用户指定变量
3. **添加注释说明**：使用用户指定变量时应添加注释说明原因
4. **避免重复定义**：避免与配置文件中的数据重复定义

#### 用户指定变量区域的处理规则

| 用户意图 | 区域内容 | 默认值位置 |
|----------|----------|------------|
| 明确指定变量 | 定义具体变量 | 可引用用户指定变量 |
| 未指定变量 | 标注 `# （用户未指定变量）` | 直接在函数参数中提供 |

#### 标准格式

**有用户指定变量时：**
```python
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
USER_TITLE_SIZE = 16
# *********************************
```

**无用户指定变量时：**
```python
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
# （用户未指定变量）
# *********************************
```

#### ⚠️ 严格禁止

- ❌ 用户未指定时在区域内添加具体变量
- ❌ 将函数默认参数放在用户指定变量区域
- ❌ 使用 `DEFAULT_` 前缀的模块级变量作为函数默认值
- ❌ 省略用户指定变量区域

### 1.3 如实测试

- **真实环境**：使用真实的配置和参数
- **完整测试**：测试被测模块的所有公开接口
- **无干扰**：不添加额外的UI重置、布局调整等干扰代码
- **可复现**：测试结果可以稳定复现
- **禁止重置**：不得使用任何参数的重置
- **调试逻辑严格限制**：调试逻辑必须使用标准格式，只能更改被测模块名称，禁止任何形式的包装、配置或参数修改

**标准格式（强制）：**
```python
# *** 调试逻辑 ***
if __name__ == "__main__":
    ft.run(lambda page: page.add(XXX.create()))
```

**严格禁止：**
- ❌ 使用 `def main(page):` 包装函数
- ❌ 设置 `page.title`, `page.padding` 等页面属性
- ❌ 传入硬编码的非默认参数
- ❌ 创建额外的回调函数包装
- ❌ 在调试逻辑中添加初始化代码（如 `配置 = 界面配置()`）
- ❌ 任何其他代码修改

**依赖初始化处理规则：**

如果被测模块依赖外部初始化（如 `ThemeProvider`），**必须在 `create` 方法内部处理**：

```python
@staticmethod
def create(...) -> ft.Container:
    # 内部处理必要的初始化
    配置 = 界面配置()
    ThemeProvider.initialize(配置)
    ...
```

**禁止在调试逻辑中处理依赖初始化：**
```python
# ❌ 错误：在调试逻辑中初始化
if __name__ == "__main__":
    配置 = 界面配置()
    ThemeProvider.initialize(配置)
    ft.run(lambda page: page.add(XXX.create()))

# ✅ 正确：在 create 方法内部初始化
if __name__ == "__main__":
    ft.run(lambda page: page.add(XXX.create()))
```

### 1.4 模块隔离

#### 调用者责任
1. 不直接创建被调用模块的内容
2. 不覆盖被调用模块的计算结果
3. 从被调用模块返回的容器中获取尺寸信息
4. 只通过公开接口访问模块
5. 避免访问私有属性（`_`前缀）
6. 通过参数传递覆盖被调模块的默认值

#### 被调用者责任
1. 完全控制自己的显示
2. 返回包含尺寸信息的容器
3. 内部处理所有必要的初始化
4. 明确标记公开接口
5. 将内部实现标记为私有
6. 返回数据副本而非内部引用

#### 模块隔离在配置管理中的应用

**核心原则：**
- **禁止往前覆盖**：调用方不能直接修改被调用方的用户指定变量
- **通过参数传递**：调用方通过函数参数传递配置，覆盖被调用方的默认行为
- **模块独立**：每个模块的用户指定变量仅在本模块内生效

**配置传递规则：**

| 场景 | 调用方行为 | 被调用方行为 | 结果 |
|------|---------|----------|------|
| **不传参数** | 使用自己的用户指定变量 | 使用自己的用户指定变量 | 各自独立 |
| **传参数** | 通过参数传递配置 | 使用传入的参数 | 覆盖默认行为 |
| **往前覆盖** | 直接修改被调用方的用户指定变量 | 用户指定变量被覆盖 | ❌ 违反模块隔离 |

**多层调用场景：**

```
组件层：设置卡片 (USER_TEXT_SIZE = 20)
    ↓ 调用（不传参数）
单元层：容器标题 (USER_TEXT_SIZE = 16)
    ↓ 调用（不传参数）
零件层：文本标签 (USER_TEXT_SIZE = 14)
```

**正确行为：**
- 设置卡片使用 `USER_TEXT_SIZE = 20`
- 容器标题使用 `USER_TEXT_SIZE = 16`
- 文本标签使用 `USER_TEXT_SIZE = 14`
- 每个模块使用自己的用户指定变量

**错误行为（往前覆盖）：**
- 设置卡片覆盖容器标题的 `USER_TEXT_SIZE = 16` 为 20
- 容器标题覆盖文本标签的 `USER_TEXT_SIZE = 14` 为 20
- 文本标签最终使用 `USER_TEXT_SIZE = 20`
- 违反模块隔离原则，影响其他调用者

**配置传递的正确方式：**

```python
# 调用方模块
@staticmethod
def create() -> ft.Container:
    # ✅ 正确：通过参数传递配置
    icon = ContainerIcon.create(icon_size=USER_ICON_SIZE)
    
    # ✅ 正确：使用被调用方的默认配置
    icon2 = ContainerIcon.create()
    
    # ❌ 错误：期望往前覆盖被调用方的用户指定变量
    # icon3 = ContainerIcon.create()  # 不能覆盖被调用方的 USER_ICON_SIZE
```

#### 参数传递覆盖规则（详细说明）

**核心概念：参数传递覆盖**

参数传递覆盖是指调用方通过函数参数传递配置值，覆盖被调用方的默认配置行为。这是在保证模块隔离保护的前提下，实现调用方个性化需求的正确方式。

**参数传递覆盖的本质：**
- **不是修改**：不修改被调用方的用户指定变量
- **不是覆盖**：不覆盖被调用方的模块级配置
- **而是传递**：通过函数参数传递运行时配置，实现个性化需求

**参数传递覆盖的三种方式：**

| 方式 | 说明 | 适用场景 | 示例 |
|------|------|---------|------|
| **使用调用方变量** | 传递调用方的用户指定变量 | 调用方有统一的个性化需求 | `Dropdown.create(width=USER_DROPDOWN_WIDTH)` |
| **使用硬编码值** | 直接传递具体数值 | 临时调整或一次性需求 | `Dropdown.create(width=100)` |
| **使用配置文件值** | 传递从配置文件获取的值 | 需要全局配置管理 | `Dropdown.create(width=配置.获取尺寸("下拉框", "width"))` |

**调用方参数传递规范：**

```python
# 方式1：使用调用方的用户指定变量（推荐）
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
# 建筑界面需要更紧凑的下拉框宽度
USER_DROPDOWN_WIDTH = 100
# *********************************

@staticmethod
def create() -> ft.Container:
    dropdown = Dropdown.create(
        options=["住宅", "商业", "工业"],
        width=USER_DROPDOWN_WIDTH  # 传递调用方的用户指定变量
    )
    return ft.Container(content=dropdown)

# 方式2：使用硬编码值（适用于临时调整）
@staticmethod
def create() -> ft.Container:
    dropdown = Dropdown.create(
        options=["选项A", "选项B"],
        width=100  # 直接传递具体数值
    )
    return ft.Container(content=dropdown)

# 方式3：使用配置文件值（适用于全局配置管理）
@staticmethod
def create() -> ft.Container:
    配置 = 界面配置()
    dropdown = Dropdown.create(
        options=["选项A", "选项B"],
        width=配置.获取尺寸("下拉框", "width_small")  # 从配置文件获取
    )
    return ft.Container(content=dropdown)
```

**被调用方参数接收规范：**

```python
# 被调用模块：下拉框.py
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
USER_WIDTH = 150  # 默认宽度，适用于大部分场景
USER_HEIGHT = 32
# *********************************

class Dropdown:
    @staticmethod
    def create(
        options: List[str] = None,
        value: str = None,
        width: int = None,  # 接收运行时参数（优先级最高）
        height: int = None,
        on_change: Callable[[str], None] = None,
        enabled: bool = True,
    ) -> ft.Container:
        # 参数优先级处理
        # 优先级：运行时参数 > 用户指定变量 > 配置文件 > 代码默认值
        
        # 1. 检查运行时参数
        current_width = width if width is not None else USER_WIDTH
        current_height = height if height is not None else USER_HEIGHT
        
        # 2. 如果需要，可以进一步检查配置文件
        # if current_width is None:
        #     配置 = 界面配置()
        #     current_width = 配置.获取尺寸("下拉框", "width_medium")
        
        # 3. 使用最终确定的值
        # ... 创建下拉框组件
```

**实际场景示例：下拉框宽度调整**

```python
# 场景：建筑界面需要更小的下拉框宽度

# 被调用模块：下拉框.py
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
USER_WIDTH = 150  # 默认宽度，适用于大部分场景
USER_HEIGHT = 32
# *********************************

class Dropdown:
    @staticmethod
    def create(
        width: int = None,  # 允许调用方覆盖默认宽度
        ...
    ) -> ft.Container:
        # 优先级：width参数 > USER_WIDTH
        current_width = width if width is not None else USER_WIDTH
        ...

# 调用方模块：建筑界面.py
# *** 用户指定变量 - AI不得修改, 变量值必须生效 ***
# 建筑界面需要更紧凑的下拉框宽度
USER_DROPDOWN_WIDTH = 100
# *********************************

class BuildingInterface:
    @staticmethod
    def create() -> ft.Container:
        # ✅ 正确：通过参数传递覆盖下拉框的默认宽度
        dropdown = Dropdown.create(
            options=["住宅", "商业", "工业"],
            width=USER_DROPDOWN_WIDTH  # 传入 100，覆盖下拉框的 USER_WIDTH = 150
        )
        
        # 结果：
        # - 下拉框模块的 USER_WIDTH 仍然是 150（未被修改）
        # - 本次调用使用宽度 100（通过参数传递）
        # - 其他调用下拉框的地方仍然使用默认宽度 150
        # - 模块隔离得到保护
        
        return ft.Container(content=dropdown)
```

**参数传递覆盖 vs 往前覆盖对比：**

| 对比项 | 参数传递覆盖（正确） | 往前覆盖（错误） |
|--------|---------------------|-----------------|
| **操作方式** | 通过函数参数传递配置 | 直接修改被调用方的用户指定变量 |
| **被调用方变量** | 保持不变 | 被修改 |
| **模块隔离** | 保护 | 破坏 |
| **影响范围** | 仅当前调用 | 影响所有调用者 |
| **其他调用者** | 不受影响 | 受影响 |

**参数传递覆盖的执行标准：**

| 检查项 | 标准 | 违规级别 |
|--------|------|---------|
| 是否通过函数参数传递 | 必须通过函数参数传递 | 严重违规 |
| 是否修改被调用方变量 | 不得修改被调用方的用户指定变量 | 严重违规 |
| 是否影响其他调用者 | 不得影响其他调用者 | 中度违规 |
| 是否保持模块隔离 | 必须保持模块隔离原则 | 严重违规 |

**违规处理：**
- **严重违规**：直接修改被调用方的用户指定变量
- **中度违规**：通过参数传递但期望覆盖用户指定变量
- **轻微违规**：未通过公开接口访问模块

### 1.5 无冗余

- **移除未使用的变量**
- **移除未使用的导入**
- **移除重复的代码**
- **移除重复的注释**

### 1.6 错误处理

- **健壮性**：对可能的异常进行捕获和处理，确保组件不会因异常而崩溃
- **优雅降级**：当组件初始化失败时，提供合理的默认行为
- **错误隔离**：确保一个组件的错误不会影响其他组件的正常运行

**标准实践：**
```python
try:
    component.update()
except:
    pass  # 静默处理，确保组件不会崩溃
```

### 1.7 组件扩展

- **功能扩展**：通过向返回的容器对象添加方法，扩展组件的功能
- **状态管理**：提供状态设置、切换和获取的方法
- **属性访问**：允许外部代码访问和修改组件的属性

**标准实践：**
```python
container.set_state = set_state
container.toggle_state = toggle_state
container.get_state = get_state
container.set_subtitle = set_subtitle
```

### 1.8 布局计算

- **自适应布局**：根据组件的实际尺寸和位置进行布局计算
- **尺寸获取**：从子组件中获取实际尺寸信息
- **位置计算**：基于基准元素（如分割线）计算其他元素的位置

**标准实践：**
```python
title_container_width = title_container_content.width
actual_divider_width = divider.width if hasattr(divider, 'width') and divider.width is not None else 100
```

### 1.9 类型提示

- **类型标注**：为函数参数和返回值添加类型提示
- **类型检查**：使用类型提示提高代码的可读性和可维护性
- **类型一致性**：确保类型提示与实际实现一致

**标准实践：**

### 1.10 配置管理最佳实践

#### 核心原则
- **单一数据源**：配置数据应集中在配置文件中管理
- **优先使用配置文件**：默认配置应放在配置文件中
- **用户指定变量作为补充**：仅在配置文件默认值不满足时使用
- **通过参数传递配置**：调用方通过函数参数传递配置，不直接修改被调用方的用户指定变量

#### 配置管理流程

**步骤1：确定配置来源**

| 配置类型 | 使用场景 | 示例 |
|---------|---------|------|
| **全局配置** | 适用于所有模块的默认配置 | 配置文件中的 `定义尺寸` |
| **模块配置** | 仅在特定模块内生效的配置 | 模块中的 `USER_XXX` 变量 |
| **运行时配置** | 运行时传入的参数，具有最高优先级 | 函数调用时传入的参数 |

**步骤2：配置优先级判断**

```python
@staticmethod
def create(
    icon_size: int = None,  # 运行时配置（优先级1）
) -> ft.Container:
    # 优先级：运行时参数 > 用户指定变量 > 配置文件 > 代码默认值
    
    # 1. 检查运行时参数
    if icon_size is None:
        # 2. 检查用户指定变量
        icon_size = USER_ICON_SIZE
        
        # 3. 检查配置文件
        if icon_size is None:
            配置 = 界面配置()
            icon_size = 配置.获取尺寸("图标", "size_medium")
        
        # 4. 使用代码默认值
        if icon_size is None:
            icon_size = 20
    
    return ft.Container(...)
```

**步骤3：配置传递规范**

| 调用方式 | 说明 | 示例 |
|---------|------|------|
| **不传参数** | 使用被调用方的默认配置 | `ContainerIcon.create()` |
| **传参数** | 覆盖被调用方的默认行为 | `ContainerIcon.create(icon_size=30)` |
| **往前覆盖** | ❌ 禁止：直接修改被调用方的用户指定变量 | - |

#### 配置变更管理

**变更流程：**
1. **评估影响**：修改配置前评估对其他模块的影响

3. **更新用户指定变量**：仅在必要时更新用户指定变量
