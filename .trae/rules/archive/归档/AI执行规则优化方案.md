# AI执行规则优化方案

## 一、现状分析

### 1.1 规则体系现状

**当前规则体系结构：**
- 核心规范：`AI执行标准_核心原则.md`（alwaysApply: true）
- 详细规范：各分类目录下的详细规范文件
- 流程规范：`开发流程_*.md`系列文件
- 检查清单：`检查清单_*.md`系列文件

**主要问题：**
1. **规则重复**：核心原则在多个文件中重复出现
2. **引用不一致**：部分#Rule引用路径存在差异
3. **歧义点**：部分规则表述存在理解歧义
4. **执行标准**：部分规则的执行标准不够明确
5. **校验机制**：缺乏有效的规则执行校验机制

### 1.2 具体问题分析

| 问题类别 | 具体表现 | 影响程度 | 优先级 |
|---------|---------|---------|--------|
| 规则重复 | 核心原则在多个文件中重复定义 | 高 | 高 |
| 引用不一致 | #Rule引用路径格式不统一 | 中 | 中 |
| 歧义点 | 部分规则表述模糊，容易产生不同理解 | 高 | 高 |
| 执行标准 | 缺乏明确的违规判定标准 | 高 | 高 |
| 校验机制 | 无自动化规则执行校验工具 | 中 | 中 |

## 二、优化方案

### 2.1 核心优化策略

**策略1：建立单一权威来源**
- 将核心原则集中到`AI执行标准_核心原则.md`
- 其他文件通过#Rule引用核心原则
- 确保规则定义的唯一性

**策略2：统一规则引用机制**
- 标准化#Rule引用格式
- 建立规则引用映射表
- 确保引用路径的一致性

**策略3：明确规则执行标准**
- 为每条规则制定明确的执行标准
- 提供详细的违规判定标准
- 建立三级违规处理机制

**策略4：建立规则执行校验机制**
- 开发规则检查脚本
- 实现自动化规则执行校验
- 提供实时规则执行反馈

### 2.2 具体改进措施

#### 2.2.1 清理规则重复

**实施步骤：**
1. **识别重复内容**：分析所有规则文件，识别重复的核心原则
2. **集中核心原则**：将核心原则集中到`AI执行标准_核心原则.md`
3. **替换引用**：将其他文件中的重复内容替换为#Rule引用
4. **验证引用**：确保所有引用路径正确

**验证方法：**
- 搜索所有规则文件中的核心原则重复内容
- 检查引用路径的正确性
- 运行规则加载测试

**验收标准：**
- 核心原则只在`AI执行标准_核心原则.md`中定义
- 其他文件通过#Rule引用核心原则
- 所有引用路径正确无误

#### 2.2.2 统一引用机制

**实施步骤：**
1. **制定引用标准**：统一#Rule引用格式为`#Rule:目录/文件名`
2. **更新引用路径**：修改所有规则文件中的引用路径
3. **建立引用映射**：创建规则引用映射表
4. **验证引用**：确保所有引用路径正确

**验证方法：**
- 检查所有规则文件中的引用格式
- 测试引用路径的可访问性
- 验证引用解析的正确性

**验收标准：**
- 所有规则文件使用统一的引用格式
- 所有引用路径正确可访问
- 引用解析无错误

#### 2.2.3 明确执行标准

**实施步骤：**
1. **分析规则歧义**：识别规则中的歧义点
2. **明确执行标准**：为每条规则制定明确的执行标准
3. **制定违规标准**：建立三级违规判定标准
4. **提供示例**：为每条规则提供详细的正确/错误示例

**验证方法：**
- 专家评审规则执行标准
- 测试规则执行的一致性
- 验证违规判定的准确性

**验收标准：**
- 每条规则都有明确的执行标准
- 违规判定标准清晰可操作
- 示例覆盖所有常见场景

#### 2.2.4 建立校验机制

**实施步骤：**
1. **开发检查脚本**：创建规则执行检查脚本
2. **集成到开发流程**：将检查脚本集成到CI/CD流程
3. **提供实时反馈**：实现规则执行的实时反馈机制
4. **建立检查清单**：创建详细的规则执行检查清单

**验证方法：**
- 运行检查脚本测试规则执行
- 验证CI/CD集成的有效性
- 测试实时反馈机制

**验收标准：**
- 检查脚本能够准确检测规则执行情况
- CI/CD集成正常工作
- 实时反馈机制有效

## 三、规则执行准确性优化

### 3.1 核心原则优化

#### 3.1.1 模块完整性

**原规则：**
> 新建模块必须包含完整的5个部分，顺序不得更改

**优化后：**
> 新建模块必须包含完整的5个部分，顺序固定为：
> 1. 文件头（编码声明+文档字符串）
> 2. 导入（标准库→第三方库→项目内部）
> 3. 用户指定变量（必须存在，不可省略）
> 4. 类/函数定义（核心功能实现）
> 5. 调试逻辑（标准化测试入口）

**执行标准：**
- 缺失任何部分 → 严重违规
- 顺序错误 → 中度违规
- 格式不正确 → 轻微违规

#### 3.1.2 用户变量保护

**原规则：**
> AI不得修改、删除或添加用户指定变量区域的任何内容

**优化后：**
> AI必须严格保护用户指定变量区域：
> - 区域必须存在，格式正确
> - 不得修改区域内任何内容
> - 不得在同一模块内二次覆盖用户指定变量
> - 用户指定的变量值必须在代码中实际使用

**执行标准：**
- 区域缺失 → 严重违规
- 修改区域内容 → 严重违规
- 二次覆盖变量 → 中度违规
- 变量值未使用 → 轻微违规

#### 3.1.3 如实测试

**原规则：**
> 调试逻辑必须使用标准格式，只能更改被测模块名称

**优化后：**
> 调试逻辑必须严格使用以下标准格式：
> ```python
> # *** 调试逻辑 ***
> if __name__ == "__main__":
>     ft.run(lambda page: page.add(XXX.create()))
> ```
> - 只能替换`XXX`为被测模块名称
> - 不得添加任何其他代码
> - 不得设置页面属性
> - 不得传入非默认参数

**执行标准：**
- 格式错误 → 严重违规
- 添加额外代码 → 中度违规
- 设置页面属性 → 中度违规
- 传入非默认参数 → 轻微违规

#### 3.1.4 模块隔离

**原规则：**
> 调用者责任：不直接创建被调用模块的内容

**优化后：**
> 调用者必须严格遵守：
> 1. 只通过公开接口访问模块
> 2. 不直接实例化被调用模块的内部类
> 3. 不直接修改被调用模块的内部数据
> 4. 不覆盖被调用模块的计算结果
> 5. 避免访问私有属性（`_`前缀）
> 6. 通过参数传递覆盖默认值

**执行标准：**
- 访问私有属性 → 严重违规
- 直接修改内部数据 → 严重违规
- 覆盖计算结果 → 中度违规
- 未通过公开接口访问 → 中度违规

#### 3.1.5 模块复用优先

**原规则：**
> 任何代码操作前，必须先检查是否存在可复用的现有模块

**优化后：**
> 任何代码操作前，必须严格执行：
> 1. **全局搜索**：搜索项目内所有相关模块
> 2. **评估匹配度**：分析功能匹配程度
> 3. **优先复用**：
>    - 功能完全匹配 → 必须复用
>    - 功能相似 → 优先通过参数调整复用
>    - 需要组合 → 创建组合层调用现有模块
>    - 完全无匹配 → 才允许创建新模块

**执行标准：**
- 未搜索现有模块 → 严重违规
- 有相似模块未复用 → 中度违规
- 复用方式不够优雅 → 轻微违规

### 3.2 流程规范优化

#### 3.2.1 新建模块流程

**优化后流程：**
1. **步骤0**：模块复用评估（必须先执行）
   - 搜索现有模块
   - 评估复用性
   - 记录决策理由
2. **步骤1**：确定模块定位
   - 确定模块层级
   - 确定职责边界
   - 确定公开接口
3. **步骤2**：创建模块结构
   - 按照标准结构创建
   - 优先复用现有模块
   - 确保模块隔离
4. **步骤3**：测试验证
   - 使用标准调试逻辑
   - 测试所有公开接口
   - 验证模块隔离

**执行标准：**
- 未执行步骤0 → 严重违规
- 模块结构不符合标准 → 中度违规
- 未优先复用现有模块 → 中度违规
- 调试逻辑不符合标准 → 轻微违规

#### 3.2.2 代码审查流程

**优化后流程：**
1. **步骤1**：模块复用审查
   - 检查是否搜索现有模块
   - 检查是否存在功能重复
   - 检查是否正确复用
2. **步骤2**：模块结构审查
   - 检查文件头和文档字符串
   - 检查导入顺序和格式
   - 检查用户指定变量区域
   - 检查类/函数定义
   - 检查调试逻辑
3. **步骤3**：模块隔离审查
   - 检查调用者责任
   - 检查被调用者责任
   - 检查数据传递方式
4. **步骤4**：代码质量审查
   - 检查函数长度
   - 检查命名规范
   - 检查代码冗余
5. **步骤5**：测试验证
   - 运行调试逻辑
   - 验证功能正确性
   - 检查运行时错误

**执行标准：**
- 未执行模块复用审查 → 严重违规
- 模块结构不符合规范 → 中度违规
- 违反模块隔离原则 → 中度违规
- 代码质量问题 → 轻微违规
