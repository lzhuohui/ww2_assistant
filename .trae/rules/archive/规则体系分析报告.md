# .trae\rules规则体系分析报告

## 一、Trae IDE规则官方文档关键技术信息

### 1.1 规则类型

| 规则类型 | 作用范围 | 适用场景 |
|---------|---------|---------|
| **个人规则** | 所有项目 | 语言风格、操作系统偏好、内容深度、交互方式 |
| **项目规则** | 当前项目 | 代码风格、语言与框架、API限定 |

### 1.2 项目规则生效方式

| 生效方式 | alwaysApply | 配置属性 | 优先级 |
|---------|------------|---------|--------|
| **始终生效** | true | globs | 中等 |
| **指定文件生效** | false | globs | 低 |
| **智能生效** | false | description | 低 |
| **手动触发生效** | false | - | 最高 |

### 1.3 规则引用机制

- **#Rule引用优先级最高**：手动触发 > 始终生效 > 智能匹配 > 文件匹配
- **相对路径**：使用相对于项目根目录的相对路径
- **历史对话**：新建或修改规则后，建议开启全新对话

### 1.4 兼容文件

| 文件 | 作用 | 配置方式 |
|------|------|---------|
| **AGENTS.md** | 项目级规则，轻量级Markdown | 设置 > 规则和技能 > 导入设置 |
| **CLAUDE.md** | Claude Code项目规则 | 设置 > 规则和技能 > 导入设置 |
| **CLAUDE.local.md** | Claude Code本地规则 | 设置 > 规则和技能 > 导入设置 |

### 1.5 最佳实践

1. **控制粒度**：单条规则保持清晰、聚焦、易于理解
2. **避免冲突**：各条规则之间不得彼此冲突或相互覆盖
3. **相对路径**：使用相对于项目根目录的相对路径
4. **优先选择**：引用规则时，优先选择与当前对话或任务强相关的规则
5. **新对话**：新建或修改规则后，建议开启全新的对话
6. **重构场景**：若项目中已有大量不符合规范的代码，明确说明当前任务为"重构"

### 2.2 模块化设计理念

#### 2.2.1 层次结构

1. **单元模块层**：基础UI组件（按钮、图标、文本等）
2. **组件模块层**：复合UI组件（卡片、导航按钮等）
3. **界面模块层**：功能界面（用户界面、策略界面等）
4. **核心接口层**：核心接口（主题、配置、状态等）
5. **配置层**：配置文件（主题配置、界面配置等）

#### 2.2.2 模块隔离原则

1. **不直接创建被调用模块的内容**
2. **不覆盖被调用模块的计算结果**
3. **用户指定变量除外**

#### 2.2.3 模块结构规范

1. **文件头**：编码声明和文档字符串
2. **导入**：必要的模块导入
3. **用户指定变量**：集中管理，AI不得修改
4. **类/函数定义**：核心功能实现
5. **调试逻辑**：标准化测试入口

## 三、现有rules体系中的缺失或冲突点

### 3.1 内容重复问题

| 问题类型 | 具体表现 | 影响程度 | 解决方案 |
|---------|---------|---------|---------|
| **主文件重复** | 代码结构规范.md和代码结构规范_精简版.md内容重复 | 高 | 删除主文件，保留精简版作为核心规范 |
| **目录重复** | Flet前端规范.md和Flet前端规范/目录并存 | 中等 | 删除主文件，保留目录结构 |
| **配置重复** | 多个alwaysApply: true规则可能冲突 | 高 | 减少alwaysApply规则数量，使用智能生效 |

### 3.2 结构混乱问题

| 问题类型 | 具体表现 | 影响程度 | 解决方案 |
|---------|---------|---------|---------|
| **层次不清** | 主文件和子目录并存，层次不清晰 | 中等 | 统一使用分层架构 |
| **引用复杂** | 规则间引用路径不统一 | 中等 | 建立统一的引用机制 |
| **命名不一致** | 使用不同分隔符（-和_） | 低 | 统一使用_分隔符 |

### 3.3 缺失的规范

| 缺失规范 | 重要性 | 影响范围 | 建议补充 |
|---------|--------|---------|---------|
| **模块化开发流程规范** | 高 | 所有开发任务 | 创建模块化开发流程规范 |
| **技能使用规范** | 中等 | AI技能调用 | 创建技能使用规范 |
| **AGENTS.md配置** | 中等 | 项目级规则 | 创建AGENTS.md文件 |
| **Flet前端开发规范** | 高 | Flet前端开发 | 完善Flet前端规范内容 |
| **测试规范** | 中等 | 测试任务 | 完善测试规范内容 |
| **文档规范** | 低 | 文档编写 | 创建文档规范 |

### 3.4 配置冲突问题

| 冲突类型 | 具体表现 | 影响程度 | 解决方案 |
|---------|---------|---------|---------|
| **alwaysApply冲突** | 多个alwaysApply: true规则可能冲突 | 高 | 减少alwaysApply规则数量 |
| **description冲突** | 多个规则description相似导致误判 | 中等 | 优化description提高匹配精度 |
| **globs冲突** | 多个规则globs重叠导致重复加载 | 中等 | 优化globs避免重叠 |

### 3.5 引用机制问题

| 问题类型 | 具体表现 | 影响程度 | 解决方案 |
|---------|---------|---------|---------|
| **引用路径不统一** | 规则间引用路径格式不一致 | 中等 | 建立统一的引用格式 |
| **引用路径错误** | 部分引用路径不存在 | 高 | 检查并修正所有引用路径 |
| **引用缺失** | 核心规范中缺少详细规范引用 | 中等 | 在核心规范中添加详细规范引用 |

### 4.3 开发流程标准

#### 4.3.1 新建模块流程

1. **检查现有模块**：先检查是否有可复用的现有模块
2. **优先复用**：优先复用已有模块，避免重复开发
3. **创建新模块**：需要时创建新模块，遵循模块结构标准
4. **测试验证**：使用调试逻辑验证模块功能

#### 4.3.2 模块调用流程

1. **导入模块**：从正确的路径导入模块
2. **调用接口**：只通过公开接口访问模块
3. **参数传递**：通过参数传递覆盖被调模块的默认值
4. **获取结果**：从被调用模块返回的容器中获取尺寸信息

1. **检查模块结构**：使用检查清单验证模块结构
