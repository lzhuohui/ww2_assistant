# Git备份提交规范

## 核心原则

- **阶段性确认**：完成阶段性工作后询问用户是否提交
- **用户主导**：AI不自动提交，由用户明确确认
- **清晰历史**：提交信息应准确描述修改内容
- **安全优先**：防止代码丢失，同时避免频繁提交

## 提交触发条件

### 必须询问用户是否提交

| 触发条件 | 说明 |
|----------|------|
| 完成单个模块的完整修复 | 修复了模块的所有规范问题 |
| 完成一组相关模块的修复 | 如修复了3个以上相关模块 |
| 累计修改超过5个文件 | 修改量较大，建议备份 |
| 修改涉及核心文件 | 配置文件、接口文件、入口文件 |
| 用户明确要求提交 | 用户说"提交"、"备份"、"git"等 |

### 不主动提交的情况

| 情况 | 说明 |
|------|------|
| 单个文件的微小修改 | 修改量小，风险低 |
| 中间调试过程 | 代码可能不完整 |
| 用户未确认的修改 | 需要用户明确同意 |
| 修改未验证通过 | 代码存在错误 |

## 提交流程

### 标准流程

```
1. AI完成阶段性修改
2. 运行验证确保代码正确
3. 向用户展示修改摘要
4. 询问："已完成X个文件的修改，是否需要提交git备份？"
5. 用户确认后执行提交
```

### 询问格式

```
已完成以下修改：
- 文件1：修复了XXX问题
- 文件2：修复了XXX问题
- ...

是否需要提交git备份？
```

## 提交信息规范

### 格式要求

```
<类型>: <简短描述>

- 详细修改项1
- 详细修改项2
- ...
```

### 类型定义

| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | feat: 添加用户登录模块 |
| fix | 修复问题 | fix: 修复输入框参数格式问题 |
| refactor | 重构代码 | refactor: 重构按钮模块结构 |
| docs | 文档更新 | docs: 更新规范文档 |
| style | 代码格式 | style: 统一代码缩进格式 |

### 示例

```
fix: 修复多个单元模块的规范合规问题

- 修复输入框模块：文档字符串格式、参数格式
- 修复通用按钮模块：导入顺序、弃用API
- 修复用户界面模块：用户变量区域、调试逻辑
```

## 禁止行为

- ❌ 未经用户确认自动提交
- ❌ 提交未验证通过的代码
- ❌ 使用模糊的提交信息（如"update"、"fix"）
- ❌ 将无关修改混合在一次提交中
- ❌ 提交包含敏感信息的文件

## 特殊情况处理

### 会话结束前

如果会话即将结束且有未提交的修改：
```
提醒用户："本次会话已完成X个文件的修改，建议提交备份后再结束会话。"
```

### 修改量过大

如果单次修改超过10个文件：
```
建议用户："本次修改涉及较多文件，建议分批提交以便回溯。"
```

### 用户长时间未回复

不采用超时自动提交机制，原因：
1. 会话可能已断开，无法执行git命令
2. 未完成的修改被提交可能导致问题
3. 用户可能有其他安排
