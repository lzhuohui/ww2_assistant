# AI被调保护规范-实现

## 规则9：被调用模块保护原则（续）

### 实现步骤

#### 第1步：当前模块适配（首选）
在不修改被调用模块的情况下解决问题：
- 数据转换：在调用前转换数据格式
- 结果处理：在接收结果后处理异常
- 条件判断：添加前置条件检查

#### 第2步：包装器模式（备选）
通过包装器扩展功能，不修改原模块

#### 第3步：继承扩展（备选）
通过继承扩展功能

#### 第4步：模块克隆调用（最后手段）
**触发条件**：以上3种方案均无法解决，且必须修改被调用模块内部逻辑

**克隆流程**：
1. 克隆被调用模块
2. 当前模块切换至新版
3. 测试验证
4. 实际运行测试其他调用方
5. 双版本并存决策
6. 长期维护与迁移

### 违规后果
- 其他调用该模块的功能出错
- 测试通过的模块失效
- 进入死循环修改

### 自检清单
- [ ] 我是否只修改了当前模块？
- [ ] 我是否查看了当前模块的所有import？
- [ ] 我是否意外修改了import的模块？

### 强制检查点

#### 修改代码前
1. 列出当前模块的所有import
2. 确认要修改的代码不在被调用模块中
3. 确认修改不会影响被调用模块的行为

#### 修改代码时
1. 如果发现需要修改被调用模块，立即停止
2. 评估是否可以使用适配方案
3. 如果必须修改，考虑克隆方案

#### 修改代码后
1. 检查是否意外修改了被调用模块
2. 检查是否硬编码了被调用模块的默认值
3. 检查是否覆盖了被调用模块的变量

### 常见违规案例

#### 案例1：硬编码默认值
```python
# 错误做法
width = setting.get("width", 150)  # 硬编码了被调用模块的默认值

# 正确做法
width = setting.get("width")  # 让被调用模块使用自己的默认值
```

#### 案例2：修改被调用模块
```python
# 错误做法：修改被调用模块
# 在被调用模块中添加新参数
def create_dropdown(width=150):  # 修改了被调用模块
    ...

# 正确做法：在调用方适配
# 在调用方传递参数
dropdown = create_dropdown(width=user_specified_width)  # 不修改被调用模块
```

#### 案例3：覆盖被调用模块的变量
```python
# 错误做法
DEFAULT_WIDTH = 150  # 覆盖了被调用模块的变量

# 正确做法
# 不定义同名变量，让被调用模块使用自己的变量
```
