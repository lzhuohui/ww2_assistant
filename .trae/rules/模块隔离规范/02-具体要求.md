# 模块隔离规范 - 具体要求

## 1. 命名规范
- **公开成员**：使用正常命名（如`process_data`）
- **私有成员**：使用下划线前缀（如`_internal_cache`）
- **保护成员**：使用双下划线前缀（如`__private_method`）

## 2. 访问控制
- **禁止访问**：调用方不得访问以下内容：
  - 私有属性和方法（`_`前缀）
  - 保护属性和方法（`__`前缀）
  - 内部模块变量（非公开导出）

## 3. 数据处理
- **返回策略**：
  - 不可变数据：可以直接返回
  - 可变数据：返回副本或深拷贝
  - 集合类型：返回新的集合实例

## 4. 依赖管理
- **依赖声明**：明确声明模块的外部依赖
- **依赖注入**：优先使用依赖注入模式
- **依赖解耦**：通过接口抽象减少模块间耦合

## 5. 违反处理

### 5.1 轻微违规
- **表现**：访问非关键私有成员
- **处理**：警告并建议修正

### 5.2 中度违规
- **表现**：直接修改被调模块的内部状态
- **处理**：提示修正并重新检查

### 5.3 严重违规
- **表现**：破坏模块隔离性，修改核心私有数据
- **处理**：拒绝执行并要求重新实现

## 6. 示例

### 6.1 正确示例
```python
# ✅ 正确：通过公开接口访问
class UserService:
    def __init__(self):
        self._users = []  # 私有属性
    
    # 公开接口
    def add_user(self, user):
        self._users.append(user)
    
    def get_users(self):
        return self._users.copy()  # 返回副本

# 调用方使用
user_service = UserService()
user_service.add_user({"name": "张三"})  # ✅ 通过公开接口
users = user_service.get_users()  # ✅ 通过公开接口
```

### 6.2 错误示例
```python
# ❌ 错误：直接访问私有属性
user_service = UserService()
user_service._users.append({"name": "李四"})  # ❌ 直接访问私有属性

# ❌ 错误：修改返回的内部引用
users = user_service.get_users()
users.append({"name": "王五"})  # 这不会影响内部数据（因为返回的是副本）

# ❌ 错误：如果返回的是引用
class BadService:
    def get_users(self):
        return self._users  # ❌ 返回内部引用

bad_service = BadService()
users = bad_service.get_users()
users.append({"name": "赵六"})  # ❌ 直接修改内部数据
```

## 7. 最佳实践

1. **明确接口**：为每个模块定义清晰的公开接口
2. **文档化**：在文档字符串中明确标记公开接口
3. **测试隔离**：为每个模块编写独立的单元测试
4. **依赖管理**：使用依赖注入框架管理模块依赖
5. **代码审查**：定期进行模块隔离性代码审查
6. **使用检查器**：使用模块隔离检查器自动检测问题

## 8. 检查要点

- [ ] 模块是否有明确的公开接口
- [ ] 私有成员是否使用`_`前缀
- [ ] 公开方法是否返回数据副本
- [ ] 调用方是否只通过公开接口访问
- [ ] 模块间是否存在循环依赖
- [ ] 依赖是否通过注入方式提供

## 9. 与模块隔离检查器的关系

- **规范是基础**：检查器的规则基于本规范制定
- **检查器是工具**：自动验证代码是否符合本规范
- **相互促进**：规范的完善可以提高检查器的准确性，检查器的使用可以发现规范中的问题
