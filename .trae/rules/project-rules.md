# Project 项目级规范总则

## 一、规范层级架构

本项目采用二级规范层级，从高到低依次为：

| 层级 | 名称 | 说明 | 约束力 |
|------|------|------|--------|
| L1 | Project 总则 | 本文件，定义核心原则和全局策略 | 最高，强制 |
| L2 | Domain 专项规范 | `专项规范/` 下的各专项规范 | 高，分类指导 |

**层级约束规则**：
- L1 规范若有冲突，L1 优先生效
- L2 规范在各自领域内优先，跨领域冲突需向 L1 追溯

## 二、优先级机制

### 2.1 规则匹配优先级（从高到低）

1. **`#Rule` 手动引用**：用户通过 `#Rule` 明确指定的规则，优先级最高
2. **`alwaysApply: true` + `globs` 匹配**：适用于当前文件的规则
3. **手动加载规则**：通过 `#Load` 或用户指定

### 2.2 冲突解决策略

当多个规则产生冲突时：

| 冲突类型 | 解决策略 |
|----------|----------|
| 数值约束冲突 | 更严格的值优先生效 |
| 流程步骤冲突 | L1 规范定义的流程优先 |
| 命名规范冲突 | 明确约定的优先于模糊的 |
| 范围重叠冲突 | 更具体的范围优先生效 |

**冲突处理流程**：
1. 检测到规则冲突时，AI 应立即停止
2. 向用户说明冲突的规则及其条款
3. 提供冲突解决建议
4. 等待用户决策后再继续

## 四、编程工作流核心框架

所有编程任务必须遵循五阶段流程：

### 4.1 阶段定义

| 阶段 | 名称 | 必须输出 | 用户确认 |
|------|------|---------|---------|
| 阶段1 | 需求澄清 | 需求确认文档 | **必须** |
| 阶段2 | 收集最佳实践 | 最佳实践总结 | 可选 |
| 阶段3 | 提供三方案 | 三个方案供选择 | **必须** |
| 阶段4 | 按规则执行 | 执行结果 | 可选 |
| 阶段5 | 核查交付 | 交付确认 | **必须** |

### 4.2 流转规则

```
阶段1 → 阶段2 → 阶段3 → 阶段4 → 阶段5
  ↓       ↓       ↓       ↓       ↓
需求     收集     提供     按规则   核查
澄清   最佳实践   三方案    执行    交付
```

- 禁止跳过任何阶段
- 禁止回退（除非用户主动要求）
- 阶段3必须等待用户选择方案

### 4.3 强制检查点

**AI必须在每次响应开头声明**：

```
【当前阶段：X - 阶段名称】
【上一阶段完成确认：是/否】
【用户确认状态：已确认/待确认】
```

**禁止行为**：
- 禁止在阶段1确认前进入阶段2
- 禁止在阶段3选择前进入阶段4
- 禁止跳过任何阶段
- 禁止省略阶段声明

### 4.4 用户交互

阶段1、阶段3、阶段5必须用户确认。

**详细要求**：见《编程工作流.md》

## 五、模块化设计核心约束（强制）

无论何时，AI 必须遵守以下模块化设计约束：

1. **禁止直接修改被调用模块**：调用者不得修改被调用模块的内部实现
2. **修改被调用者需评估影响**：如必须修改，必须先评估对其他调用者的影响
3. **修改需用户同意**：影响评估后，必须获得用户明确同意方可继续
