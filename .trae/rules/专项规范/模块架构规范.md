# 模块架构规范

## 一、模块拆分原则

### 1.1 单一职责原则
每个模块只负责一个明确的功能领域：
- 界面显示 → 脚本界面类
- 动作执行 → 游戏动作类
- 信息查询 → 游戏查询类
- 数据处理 → 数据操作类

### 1.2 接口隔离原则
模块之间通过清晰接口通信：
- 调用者不依赖不需要的接口
- 被调用者不暴露内部实现
- 接口参数明确，返回值清晰

### 1.3 依赖倒置原则
高层模块不依赖低层模块，二者都依赖抽象：
- 通过配置文件传递参数
- 通过回调函数传递控制
- 通过事件机制传递状态

## 二、模块分类判断

### 2.1 分类体系总览

本项目采用两套分类体系，互不冲突：

| 分类体系 | 适用场景 | 详细说明 |
|----------|----------|----------|
| **V2前端架构** | V2版本前端界面模块 | 本规范第2.2节 |
| **四类模块** | 通用模块分类 | 本规范第2.3节 |

### 2.2 V2前端架构层级（五层架构）

**来源**：《V2版本重构共识文档》第二章

V2前端采用五层架构设计，从上到下依次为：

```
┌─────────────────────────────────────────────────────────────┐
│                    V2五层架构                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  层级1：主入口V2                                             │
│  ─────────────                                              │
│  职责：导航按钮 + 调用功能界面                               │
│  内容：导航面板 + 内容区域                                   │
│  销毁：调用功能界面.destroy()                               │
│  模块职责：                                                 │
│  - 定义1：从配置服务获取导航列表                            │
│  - 定义2：上层只传递page和config_service                    │
│  - 定义3：theme可覆盖                                       │
│  - 定义4：create()、destroy()、switch_page()               │
│                                                             │
│  层级2：功能界面V2                                           │
│  ─────────────                                              │
│  职责：调用卡片组 + 布局                                     │
│  内容：系统界面、策略界面、建筑界面、账号界面                 │
│  销毁：调用卡片组.dropdown.destroy_all()                    │
│  模块职责：                                                 │
│  - 定义1：从配置服务获取section列表、卡片信息、控件列表     │
│  - 定义2：上层只传递config_service                          │
│  - 定义3：theme_colors可覆盖                                │
│  - 定义4：create()、destroy()                              │
│                                                             │
│  层级3：卡片组V2                                             │
│  ─────────────                                              │
│  职责：调用卡片+开关+控件                                    │
│  内容：卡片组（集成开关+控件创建+控件布局）                   │
│  销毁：提供dropdown属性访问                                 │
│  模块职责：                                                 │
│  - 定义1：组合卡片开关和卡片控件，处理默认值合并逻辑        │
│  - 定义2：上层只传递section列表                             │
│  - 定义3：theme_colors可覆盖                                │
│  - 定义4：create()、destroy_all()、dropdown属性             │
│                                                             │
│  层级4：复合模块                                             │
│  ─────────────                                              │
│  职责：组合基础模块                                          │
│  内容：图标标题、导航按钮、卡片（左侧开关+右侧控件区）        │
│  销毁：无                                                   │
│  模块职责：                                                 │
│  - 定义1：自己获取enabled状态、title、icon、subtitle等      │
│  - 定义2：上层只传递section                                 │
│  - 定义3：theme_colors可覆盖                                │
│  - 定义4：create()、get_state()、toggle()等                 │
│                                                             │
│  层级5：基础模块                                             │
│  ─────────────                                              │
│  职责：单一功能                                              │
│  内容：卡片容器、输入框、下拉框、主题色块、标签、图标、分割线 │
│  销毁：只有下拉框有销毁功能                                  │
│  模块职责：                                                 │
│  - 定义1：自己获取数据（选项列表、当前值、宽度高度等）       │
│  - 定义2：上层只传递section和control_id                     │
│  - 定义3：theme_colors、width、height等可覆盖               │
│  - 定义4：create()、get_default_value()、reset_to_default() │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**模块职责四项定义说明**：
- 定义1（组件独立功能）：模块自己获取所需数据
- 定义2（上层调用）：上层只传递定位信息，不传递数据
- 定义3（用户指定变量覆盖）：允许上层覆盖特定变量
- 定义4（公开接口）：提供完整公开接口供上层调用

**V2模块判断关键词**：
- 包含"V2"、"五层架构"、"卡片组"、"懒加载" → **V2前端模块**
- 调用《模块代码结构.md》第七节（四部分结构）

### 2.3 四类模块定义（通用分类）

| 分类 | 功能描述 | 技术栈 | 变量使用 | 代码结构 |
|------|----------|--------|----------|----------|
| **脚本界面类** | Flet可视化界面，用户交互 | flet | USER_变量为模块组成部分 | 《模块代码结构.md》三、脚本界面类 |
| **游戏动作类** | 游戏内动作执行，坐标操作 | pyautogui, time | 测试数据在if __name__内部 | 《模块代码结构.md》四、游戏动作类 |
| **游戏查询类** | 游戏信息查询，图像识别 | cv2, numpy, PIL | 测试数据在if __name__内部 | 《模块代码结构.md》五、游戏查询类 |
| **数据操作类** | 数据存储读取，配置管理 | json, os, pickle | 测试数据在if __name__内部 | 《模块代码结构.md》六、数据操作类 |

### 2.4 判断流程（AI必须执行）

**第一步：AI分析需求提取关键词**
从用户需求描述中识别关键词，用于匹配模块类型。

**第二步：按顺序匹配分类**：

1. 包含以下任一关键词 → **V2前端模块**
   - V2、五层架构、卡片组、懒加载、销毁链路
   
2. 包含以下任一关键词 → **脚本界面类**
   - 界面、窗口、UI、显示、面板、控制台、可视化
   
3. 包含以下任一关键词 → **游戏动作类**
   - 点击、滑动、动作、执行、操作、移动、拖拽、按压
   
4. 包含以下任一关键词 → **游戏查询类**
   - 查询、识别、检测、查找、搜索、状态、位置、存在
   
5. 包含以下任一关键词 → **数据操作类**
   - 保存、读取、数据、配置、设置、记录、存储、加载

**第三步：默认规则**
- 无明确关键词 → 脚本界面类
- 多个关键词匹配 → 按上述顺序优先匹配

### 2.5 执行要求
1. 只匹配一个类型
2. 调用《模块代码结构.md》中对应模板
3. **变量使用规则**：
   - V2前端模块：数据和文件接口区域为模块组成部分，AI不得修改
   - 脚本界面类：USER_变量为模块组成部分，必须生效，AI不得修改
   - 其他类型：测试数据在if __name__内部，与模块隔离

## 三、模块隔离约束

### 3.1 调用约束
1. **禁止直接修改被调用模块**：调用者不得修改被调用模块的内部实现
2. **修改被调用者需评估影响**：如必须修改，必须先评估对其他调用者的影响
3. **修改需用户同意**：影响评估后，必须获得用户明确同意方可继续

### 3.2 数据约束
1. **配置数据分离**：运行时常量使用USER_前缀变量
2. **运行时数据隔离**：模块间通过参数传递数据
3. **持久化数据独立**：文件操作使用独立数据文件

### 3.3 测试约束
1. **测试不干扰功能**：测试逻辑不得添加额外数据
2. **测试如实反映**：测试结果必须反映模块真实功能
3. **测试独立验证**：每个模块有独立的测试逻辑
