# 模块代码标准结构

## 一、结构总览

| 模块类型 | 区域数 | 结构差异 | 测试输出 |
|----------|--------|----------|----------|
| 脚本界面类 | 6区域 | 有独立指定变量区域 | 界面直接验证 |
| 游戏动作类 | 5区域 | 测试数据在测试逻辑内部 | 游戏界面直接验证 |
| 游戏查询类 | 5区域 | 测试数据在测试逻辑内部 | 测试逻辑打印结果 |
| 数据操作类 | 5区域 | 测试数据在测试逻辑内部 | 测试逻辑打印结果 |

## 二、类/函数区域接口规范

### 2.1 公开函数与内部函数

| 类型 | 命名规则 | 调用范围 | 说明 |
|------|----------|----------|------|
| 公开函数 | 无下划线前缀 | 可被其他模块调用 | 模块对外接口 |
| 内部函数 | 下划线开头（_func_name） | 仅模块内部调用 | 辅助实现 |

### 2.2 公开函数要求

- 使用类型注解明确参数和返回值
- 简短文档字符串说明功能
- 参数必须有默认值

```python
# 公开函数示例
def execute_action(click_x: int, click_y: int, delay: float = 1.0) -> bool:
    """执行点击动作，返回是否成功"""
    _validate_coords(click_x, click_y)
    pyautogui.click(click_x, click_y)
    time.sleep(delay)
    return True

# 内部函数示例
def _validate_coords(x: int, y: int) -> bool:
    """验证坐标有效性"""
    return 0 <= x <= 1920 and 0 <= y <= 1080
```

### 2.3 模块调用约束

- 调用者只能调用公开函数（无下划线前缀）
- 禁止调用内部函数（下划线开头）
- 禁止直接修改被调用模块的内部实现

## 三、脚本界面类结构（6区域）

**适用**：Flet可视化界面模块

### 区域顺序（不得更改）

```
1. 文件开头区域
2. 设计记录区域
3. 模块导入区域
4. 指定变量区域 ← USER_变量为模块组成部分，必须生效
5. 类/函数区域
6. 测试逻辑区域
```

### 完整模板

```python
# -*- coding: utf-8 -*-

"""
模块名称：[文件名称.py]  # 如：main_ui.py
模块功能：[Flet界面功能描述]
实现步骤：
- 步骤1
- 步骤2
"""

import flet as ft

# *** 用户指定变量: 变量值必须生效,AI不得更改数据 ***
USER_WINDOW_TITLE = "窗口标题"      # 主窗口标题
USER_BUTTON_TEXT = "按钮文字"       # 按钮显示文字
# *********************************

def main(page: ft.Page):
    """主界面函数"""
    page.title = USER_WINDOW_TITLE
    page.add(ft.Text(USER_WINDOW_TITLE), ft.ElevatedButton(USER_BUTTON_TEXT))

# *** 标准测试格式: 仅调用被测模块,AI不得添加数据 ***
if __name__ == "__main__":
    ft.app(target=main)
```

**说明**：
- 模块有 `main` 函数时，直接 `ft.app(target=main)`
- 模块只有组件类时：`ft.app(target=lambda p: p.add(组件名.create()))`

```python

"""
模块名称：[文件名称.py]  # 如：data_manager.py
模块功能：[数据操作功能描述]
实现步骤：
- 步骤1
- 步骤2
"""

import json
import os

def save_data(data, file_path="data.json"):
    """保存数据到文件"""
    with open(file_path, 'w', encoding='utf-8') as f:
        json.dump(data, f, ensure_ascii=False, indent=2)
    return True

def load_data(file_path="data.json", default_data=None):
    """从文件加载数据"""
    if os.path.exists(file_path):
        with open(file_path, 'r', encoding='utf-8') as f:
            return json.load(f)
    return default_data if default_data is not None else {}

# *** 标准测试格式: 仅调用被测模块,AI不得添加数据 ***
if __name__ == "__main__":
    USER_TEST_FILE = "test.json"           # 测试文件路径
    USER_TEST_DATA = {"key": "value"}      # 测试数据
    save_data(USER_TEST_DATA, USER_TEST_FILE)
    result = load_data(USER_TEST_FILE)
    print(f"读取结果: {result}")

## 七、V2前端模块代码结构（四部分结构）

**适用**：V2版本前端界面模块（五层架构）

### 7.1 结构来源

本结构依据《V2版本重构共识文档》第五章第4节定义，与V1六区域结构并存，互不冲突。

### 7.2 结构顺序（不得更改）

```
1. 模块说明区域（文档字符串）
2. 数据和文件接口区域（前置，方便查看和修改）
3. 公开接口区域

### 7.4 V2结构优势

| 优势 | 说明 |
|------|------|
| 方便查看 | 数据和文件接口在代码最前面，一目了然 |
| 方便修改 | 修改配置不需要阅读整个代码 |
| 方便调用 | 调用者可以快速了解需要传递的参数 |
| 统一风格 | 所有V2模块遵循相同的代码结构 |

**注意**：测试逻辑不能添加任何数据覆盖，只能运行被测模块/函数的内部逻辑和数据，运行结果应真实反映被测模块。

### 7.5 V2模块销毁责任

| 层级 | 模块 | 是否需要销毁方法 | 说明 |
|------|------|-----------------|------|
| 1 | 主入口V2 | 需要 | 调用功能界面.destroy() |
| 2 | 功能界面V2 | 需要 | 调用卡片组.dropdown.destroy_all() |
| 3 | 卡片组V2 | **不需要** | 提供dropdown属性访问 |
| 5 | 下拉框V2 | **需要** | 销毁懒加载菜单 |
| 5 | 输入框V2 | 不需要 | 没有需要销毁的东西 |

## 八、AI模块处理流程

```
步骤1：调用《模块架构规范.md》判断模块类型
步骤2：根据类型选择对应结构
       - V2前端模块 → 四部分结构（本规范第七节）
       - V1界面类 → 六区域结构（本规范第三节）
       - 其他类 → 五区域结构（测试数据在if __name__内部，与模块隔离）
步骤3：按模板生成/处理代码
步骤4：核查代码结构是否符合模板
步骤5：发现不一致立即修正
步骤6：V2模块保护数据和文件接口区域不被修改
```
