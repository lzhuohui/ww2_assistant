# 模块管理器

## 描述
管理模块的生命周期，确保模块一旦测试通过即冻结，防止AI反复修改已测试模块导致的死循环。

## 使用场景
- 启动新模块开发
- 冻结已测试模块
- 检查模块状态
- 管理模块依赖

## 指令

### 步骤1：模块注册
创建模块时，在 `.trae/modules/registry.md` 中注册：
```markdown
## 模块A
- 名称：module-a
- 路径：src/core/module_a.py
- 状态：开发中
- 依赖：module-b, module-c
- 契约：contracts/module-a-interface.md
```

### 步骤2：契约定义
在 `contracts/module-a-interface.md` 中定义接口契约：
```markdown
# 模块A接口契约

## 函数签名
```python
def process_data(input: Data) -> Result:
    """处理数据"""
    pass
```

## 前置条件
- input 不为 None
- input.data 长度 > 0

## 后置条件
- 返回 Result 对象
- Result.status 为 "success" 或 "error"

## 性能约束
- 时间复杂度：O(n)
- 最大执行时间：100ms
```

### 步骤3：契约审查
- 提交契约给用户审查
- 等待用户批准
- 批准后锁定契约

### 步骤4：模块实现
- 严格按照契约实现
- 不得超出契约范围
- 添加单元测试

### 步骤5：模块冻结（⚠️ 必须用户确认）

#### 冻结前准备
测试通过后，AI必须：
1. 向用户报告：
   ```
   模块 [module-a] 测试已完成
   - 测试用例：X个通过
   - 代码覆盖率：XX%
   - 依赖模块：均已冻结
   
   建议：冻结该模块（禁止后续修改）
   
   确认冻结？ [是/否]
   ```
2. 等待用户明确确认

#### 用户确认后执行
3. 更新注册表状态为"已冻结"
4. 在模块文件头部添加标记：
   ```python
   # MODULE: module-a
   # STATUS: FROZEN
   # VERSION: 1.0
   # DO NOT MODIFY
   ```
5. 创建锁定文件：`.trae/modules/locked/module-a.lock`
6. 添加Git标签：`module-a-v1.0-frozen`
7. 报告用户："模块已冻结，禁止修改"

### 步骤6：依赖检查
冻结前检查：
- 所有依赖模块是否已冻结
- 是否存在循环依赖
- 契约是否兼容

## 模块状态管理

### 状态转换
```
开发中 → 测试中 → 已冻结 → 已发布
  ↑        ↑        ↑         ↑
可修改   可修复   禁止修改  禁止修改
```

### 状态查询
查询模块状态：
```bash
@module-manager 查询模块状态 module-a
```

### 状态列表
列出所有模块状态：
```bash
@module-manager 列出所有模块
```

## 防止死循环机制

### 机制1：物理锁定
- 冻结模块文件设置为只读
- AI无法修改锁定文件

### 机制2：上下文隔离
- 冻结模块代码从AI上下文移除
- AI只能看到接口定义

### 机制3：强制报告
- 尝试修改冻结模块时，必须报告用户
- 不得擅自修改

### 机制4：被调用模块保护（最关键）
**核心原则：解决当前模块问题时，禁止修改被调用模块**

#### 识别被调用模块
在修改代码前，必须执行：
```bash
@module-manager 分析依赖 [当前模块名]
```

输出示例：
```
当前模块：module-a
状态：开发中（可修改）

被调用模块（禁止修改）：
- module-b (import from module_b)
- module-c (import module_c)
- utils.helper (import utils.helper)

调用方模块（禁止修改）：
- module-d (module-d 调用了 module-a)
```

#### 开发时约束
- 只能修改当前模块（module-a）
- 禁止修改所有import的模块
- 禁止修改调用当前模块的模块

#### 发现被调用模块问题时的自适应处理

AI必须按以下优先级尝试解决问题：

**第1步：当前模块适配（必须尝试）**
```bash
@module-manager 尝试适配 [当前模块] [问题描述]
```
策略：
- 数据转换适配
- 结果处理适配
- 添加前置检查
- 提供默认值

**第2步：包装器模式（如第1步失败）**
```bash
@module-manager 创建包装器 [被调用模块] [扩展功能]
```
策略：
- 创建适配器类
- 使用装饰器
- 实现代理模式

**第3步：继承扩展（如适用）**
```bash
@module-manager 继承扩展 [被调用模块类] [新方法]
```
策略：
- 继承原类
- 添加新方法
- 重写方法（谨慎）

**第4步：模块克隆调用（最后手段）**
⚠️ **重要**：克隆是最后手段，优先使用适配、包装、继承保持模块化设计

**触发条件**：
- 以上3步均失败
- **必须修改被调用模块内部逻辑**（如算法bug、性能问题、安全漏洞）
- 接口不匹配或功能不足应使用适配/包装/继承，不应克隆

**4.1 克隆被调用模块**
```bash
@module-manager 克隆模块 [原模块] [新模块]
# 示例：@module-manager 克隆模块 module-b module-b-v2
# 注意：仅当必须修改内部逻辑时才克隆
```
操作：
- 复制原模块代码到新文件
- 保持原模块完全不变
- 新模块标记为开发中
- 记录克隆原因（为什么必须修改内部逻辑）

**4.2 当前模块切换至新版**
- 修改当前模块的import，指向新版
- 保持其他模块不变

**4.3 测试验证**
```bash
@module-manager 测试模块 [当前模块] [新版模块]
```

**4.4 实际运行测试其他调用方**（关键）
```bash
@module-manager 实际测试调用方 [原模块]
```
- 逐一启动其他调用方模块
- 实际运行核心功能
- 验证是否受影响
- 记录实际测试结果

**4.5 双版本并存决策**
向用户提交方案：
```
方案A：立即统一更新（激进）
- 风险高，需完整测试

方案B：双版本并存（推荐）⭐
- 当前模块用新版
- 其他模块用原版
- 风险低，按需迁移
```

**4.6 长期维护**
- 维护迁移日志
- 按需逐一迁移
- 最终统一后废弃原版

**第5步：直接修改被调用模块（最后手段）**
仅当克隆方案不可行时：
```bash
@module-manager 请示修改 [被调用模块]
```

必须提交**影响预判报告**：
- 实际运行测试结果（每个调用方）
- 风险评估
- 方案对比
- 决策建议

#### 处理原则
- 必须尝试至少3种适配方案
- 详细记录每种方案的尝试过程
- 只有确认无法适配时才能请示用户
- 不得跳过适配直接要求修改

## 违规处理

### 发现违规修改
1. 立即撤销修改
2. 报告用户
3. 恢复模块到冻结状态

### 解冻流程
如需修改已冻结模块：
1. 提交解冻申请
2. 说明修改原因
3. 用户批准
4. 创建新版本（如 v1.1）
5. 重新测试并冻结
