# 被调模块保护器

## 核心原则

**未经用户明确同意，不得修改任何已测试的被调模块**

这是保护项目稳定性和可维护性的核心原则。已测试的模块是项目的稳定基石，随意修改可能导致其他调用者出错，造成连锁反应。

## 工作流程

```

├─────────────────────────────────────────────────────────────┤

│    - 如果涉及，强制触发确认流程                                 │
│    - 记录所有修改意图                                          │
├─────────────────────────────────────────────────────────────┤
│  阶段3：任务完成前 - 强制检查                                  │
```

## 阶段1：任务开始时 - 建立保护清单

### 必须执行

**1. 扫描项目目录，识别所有模块**
```
扫描范围：项目根目录下的所有.py文件
识别标准：
- 包含类或函数定义的文件
- 被其他模块导入的文件
- 带有保护标记的文件
```

**2. 读取模块文件，查找保护标记**
```
保护标记格式：
# *** 已测试模块 - 谨慎修改 ***
# 最后测试时间：YYYY-MM-DD
# 测试状态：✅ 通过 / ❌ 未测试
# 调用者数量：N个模块
# ******************************
```

**3. 建立被调模块保护清单**

**4. 向用户展示保护清单**

### 保护清单格式

```markdown
🔒 被调模块保护清单

以下模块已测试通过，未经同意不得修改：

1. 🔴 [模块A.py] - 核心模块，被3个模块调用
   测试状态：✅ 通过
   最后测试：2026-03-15
   保护级别：高
   位置：业务逻辑层/

2. 🟡 [模块B.py] - 重要模块，被1个模块调用
   测试状态：✅ 通过
   最后测试：2026-03-15
   保护级别：中
   位置：工具层/

3. 🟢 [模块C.py] - 独立模块，暂无调用者
   测试状态：✅ 通过
   最后测试：2026-03-15
   保护级别：低
   位置：组件层/

⚠️ 警告：修改以上模块可能导致其他调用者出错！

保护规则：
1. 未经用户明确同意，不得修改任何已测试模块
2. 优先选择创建新模块或扩展现有接口
3. 如果必须修改，必须经过完整的确认流程
```

### 保护级别定义

| 保护级别 | 标识 | 说明 | 触发条件 |
|---------|------|------|----------|
| 🔴 高保护 | 核心模块 | 被多个模块调用，修改影响大 | 调用者 > 3个 |
| 🟡 中保护 | 重要模块 | 被少量模块调用 | 调用者 1-3个 |
| 🟢 低保护 | 独立模块 | 暂无其他调用者 | 调用者 = 0个 |

## 阶段2：任务执行中 - 主动防护

### 每次生成代码前，必须自问

```

1. 我要修改的文件是否在保护清单中？
   ├── 是 → 继续检查
   └── 否 → 正常生成代码

2. 该文件是否带有"已测试模块"保护标记？
   ├── 是 → 立即停止，触发确认流程
   └── 否 → 正常生成代码，但建议添加保护标记

3. 我的修改是否会影响其他调用者？
   ├── 是 → 触发确认流程
   └── 否 → 记录修改意图，继续生成
```

## 保护措施

### 措施1：强制确认

**规则**：
- 任何对已测试模块的修改都必须经过用户确认
- AI不得自行决定修改被调模块
- 默认策略是保护，而非修改

**实施方式**：
- 在每次涉及被调模块的操作前强制确认
- 在提交成果前强制用户逐条确认
- 未经同意的修改无法提交

### 措施2：替代方案优先

**优先级**：
1. **创建新模块**（最高优先级）
   - 完全隔离，不影响现有模块
   - 通过依赖注入或接口调用
   
2. **扩展现有接口**
   - 添加新的公开方法
   - 不影响现有调用者
   
3. **通过参数配置**
   - 使用参数控制行为
   - 灵活，无需修改代码
   
4. **修改被调模块**（最后选择）
   - 必须经过完整确认流程
   - 必须获得用户明确同意

### 措施3：影响评估

**必须评估**：
- **调用者识别**：列出所有调用此模块的模块
- **向后兼容性**：评估是否破坏现有接口
- **测试要求**：确定需要重新测试的范围
- **风险等级**：评估修改的风险程度

### 措施4：回滚机制

**回滚策略**：
- 未经同意的修改必须回滚
- 提供一键回滚功能
- 保留修改历史记录
- 允许用户选择回滚或确认

## 违规处理

### 未经同意修改被调模块

**检测方式**：
- 对比任务开始和结束时的模块状态
- 检查修改报告中的未经同意修改
- 用户反馈或代码审查发现

**处理流程**：
1. 立即停止提交
2. 生成违规报告
3. 标记违规修改
4. 要求回滚或获得同意
5. 记录违规行为
6. 加强后续监控

### 未经过确认流程

**检测方式**：
- 检查修改意图记录
- 验证用户确认记录
- 审查确认流程完整性

**处理流程**：
1. 标记为潜在违规
2. 要求补充确认流程
3. 评估是否需要回滚
4. 完善确认记录
5. 加强后续监控

### 未评估影响范围

**检测方式**：
- 检查影响评估记录
- 验证调用者识别
- 审查测试计划

**处理流程**：

3. 准备测试计划
4. 获得用户确认
5. 更新修改报告

**检查时机**：
- 被调模块保护器：任务开始、执行中、完成前、提交前
- 模块隔离检查器：代码生成时、架构审查时

## 注意事项

1. **严格保护**：对已测试模块采取最严格的保护措施
2. **优先替代**：始终优先选择替代方案，修改是最后选择
3. **完整记录**：记录所有修改意图和用户确认
4. **强制确认**：未经用户明确同意，不得提交任何修改
5. **及时回滚**：发现未经同意的修改，立即回滚
6. **持续监控**：在整个开发过程中持续监控修改意图
7. **联动配合**：与模块化开发流程和模块隔离检查器紧密配合
