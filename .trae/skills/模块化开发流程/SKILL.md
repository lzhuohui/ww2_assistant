# 模块化开发流程

## 核心设计哲学

1. **优先复用**：开发功能时，首先检查并调用已存在的模块
2. **按需创建**：如需要的功能不存在，先生成此功能模块后再调用
3. **模块隔离**：确保模块间正确交互，调用者不得修改被调模块内部实现
4. **标准化接口**：所有模块通过统一接口进行交互和集成

## 模块开发工作流程

### 阶段1：需求分析
- 明确功能需求和边界
- 识别功能的核心职责
- 确定可能的依赖关系

### 阶段2：模块检查（关键步骤）
**必须执行以下检查**：

1. **扫描现有模块**
   - 检查项目目录结构
   - 查看已存在的功能模块
   - 阅读模块的文档字符串了解功能

2. **功能匹配评估**
   - 评估现有模块是否满足需求
   - 检查模块接口是否兼容
   - 确定是否需要扩展现有模块

3. **决策判断**
   - ✅ **如果模块存在且满足需求**：直接调用现有模块
   - ✅ **如果模块存在但需扩展**：扩展现有模块接口
   - ❌ **如果模块不存在**：创建新模块

### 阶段3：模块调用（优先路径）
**当现有模块满足需求时**：

```python
# ✅ 正确示例：调用现有模块
from 现有模块 import 功能类

# 通过公开接口调用
实例 = 功能类(配置)
结果 = 实例.公开方法(参数)
```

**调用原则**：
- 只能通过公开接口访问
- 不得修改被调模块内部实现
- 遵循模块隔离规范

### 阶段4：模块创建（当需要时）
**当现有模块不满足需求时**：

1. **创建模块文件**
   - 按照代码结构规范创建模块
   - 包含完整的5个部分结构
   - 编写清晰的文档字符串

2. **定义模块接口**
   - 明确公开接口和方法
   - 使用`_`前缀标记私有实现
   - 提供接口文档说明

3. **模块验证**
   - 确保模块结构完整
   - 验证接口定义清晰
   - 通过调试逻辑测试

4. **模块注册**
   - 更新模块索引（如有）
   - 记录模块功能和接口
   - 便于后续查找和复用

### 阶段5：模块集成
**调用新创建的模块**：


## 决策流程图

```
开始开发新功能
    ↓
分析功能需求
    ↓
检查现有模块
    ↓
是否存在满足需求的模块？
    ↓ 是          ↓ 否
调用现有模块   创建新模块
    ↓              ↓
通过公开接口   按照规范创建
调用            定义接口
    ↓              ↓
验证调用方式   验证模块结构
    ↓              ↓
完成开发      调用新模块
                  ↓
               完成开发
```

## 最佳实践

1. **先查后建**：始终先检查现有模块，避免重复开发
2. **文档优先**：确保模块文档清晰，便于后续查找和复用
3. **接口稳定**：公开接口一旦确定，尽量保持稳定
4. **隔离保护**：严格遵循模块隔离规范，保护被调模块
5. **持续维护**：定期更新模块索引，维护模块文档

## 错误示例

### ❌ 错误1：未检查直接创建
```python
# 错误：没有检查是否已存在类似功能的模块
# 直接创建了新模块，导致功能重复
class 数据处理:
    pass
```

### ❌ 错误2：直接修改被调模块
```python
# 错误：直接修改被调模块的内部实现
from 现有模块 import 功能类

实例 = 功能类()
实例._internal_data = 新数据  # ❌ 直接修改私有属性
```

### ❌ 错误3：绕过公开接口
```python
# 错误：绕过公开接口直接调用私有方法
from 现有模块 import 功能类

实例 = 功能类()
结果 = 实例._private_method()  # ❌ 调用私有方法
```

## 与现有规范的配合

- **代码结构规范**：指导模块创建的标准结构
- **模块隔离规范**：确保模块间正确交互，调用者不得修改被调模块
- **模块隔离检查器**：自动验证模块调用和创建的规范性
- **命名规范**：确保模块和接口命名的一致性

## 成功标准

- ✅ 优先复用已有模块，避免重复开发
- ✅ 新模块必须按照标准模板创建
- ✅ 所有模块交互必须通过公开接口
- ✅ 严格遵循模块隔离规范
- ✅ 模块文档清晰完整，便于后续复用
