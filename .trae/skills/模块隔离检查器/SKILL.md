# 模块隔离检查器

## 核心原则

**调用者不得更改被调模块**

这是模块化设计的核心原则，确保模块之间的独立性和可维护性。

## 检查规则

### 1. 接口隔离原则

**规则：调用者只能通过公开接口访问被调模块**

✅ **正确示例：**
```python
# 被调模块：数据处理.py
class DataProcessor:
    def process(self, data):
        """公开接口：处理数据"""
        return self._internal_process(data)
    
    def _internal_process(self, data):
        """私有方法：内部实现"""
        return data.strip()

# 调用者：业务逻辑.py
from 数据处理 import DataProcessor

processor = DataProcessor()
result = processor.process(data)  # ✅ 通过公开接口调用
```

❌ **错误示例：**
```python
# 调用者：业务逻辑.py
from 数据处理 import DataProcessor

processor = DataProcessor()
result = processor._internal_process(data)  # ❌ 直接调用私有方法
```

### 2. 数据隔离原则

**规则：调用者不得直接修改被调模块的内部数据**

✅ **正确示例：**
```python
# 被调模块：配置管理.py
class ConfigManager:
    def __init__(self):
        self._config = {"width": 100}
    
    def get_config(self):
        """公开接口：获取配置副本"""
        return self._config.copy()
    
    def set_config(self, key, value):
        """公开接口：设置配置"""
        self._config[key] = value

# 调用者：界面组件.py
from 配置管理 import ConfigManager

manager = ConfigManager()
config = manager.get_config()  # ✅ 获取副本
config["width"] = 200  # ✅ 修改副本不影响原数据
manager.set_config("width", 200)  # ✅ 通过接口修改
```

❌ **错误示例：**
```python
# 调用者：界面组件.py
from 配置管理 import ConfigManager

manager = ConfigManager()
config = manager._config  # ❌ 直接访问内部数据
config["width"] = 200  # ❌ 直接修改内部数据
```

### 3. 依赖隔离原则

**规则：调用者不得依赖被调模块的实现细节**

✅ **正确示例：**
```python
# 被调模块：日志记录.py
class Logger:
    def log(self, message):
        """公开接口：记录日志"""
        self._write_to_file(message)
    
    def _write_to_file(self, message):
        """私有方法：写入文件（实现细节）"""
        with open("log.txt", "a") as f:
            f.write(message + "\n")

# 调用者：业务逻辑.py
from 日志记录 import Logger

logger = Logger()
logger.log("操作成功")  # ✅ 只依赖公开接口
```

❌ **错误示例：**
```python
# 调用者：业务逻辑.py
from 日志记录 import Logger

logger = Logger()
# ❌ 依赖实现细节：假设日志写入log.txt
with open("log.txt", "r") as f:
    content = f.read()
```

### 4. 状态隔离原则

**规则：调用者不得直接修改被调模块的内部状态**

✅ **正确示例：**
```python
# 被调模块：用户会话.py
class UserSession:
    def __init__(self):
        self._is_active = False
    
    def login(self):
        """公开接口：登录"""
        self._is_active = True
    
    def logout(self):
        """公开接口：登出"""
        self._is_active = False
    
    def is_active(self):
        """公开接口：检查状态"""
        return self._is_active

# 调用者：认证系统.py
from 用户会话 import UserSession

session = UserSession()
session.login()  # ✅ 通过接口修改状态
if session.is_active():  # ✅ 通过接口检查状态
    pass
```

❌ **错误示例：**
```python
# 调用者：认证系统.py
from 用户会话 import UserSession

session = UserSession()
session._is_active = True  # ❌ 直接修改内部状态
```

### 5. 继承隔离原则

**规则：子类不得破坏父类的封装性**

✅ **正确示例：**
```python
# 父类：基础组件.py
class BaseComponent:
    def render(self):
        """公开接口：渲染组件"""
        return self._build_content()
    
    def _build_content(self):
        """受保护方法：构建内容"""
        return "基础内容"

# 子类：高级组件.py
from 基础组件 import BaseComponent

class AdvancedComponent(BaseComponent):
    def _build_content(self):
        """重写受保护方法"""
        return "高级内容"
```

❌ **错误示例：**
```python
# 子类：高级组件.py
from 基础组件 import BaseComponent

class AdvancedComponent(BaseComponent):
    def render(self):
        """❌ 重写公开接口，破坏父类契约"""
        return "完全不同的内容"
```

## 检查清单

### 调用者检查项

- [ ] 是否只通过公开接口访问被调模块？
- [ ] 是否避免直接访问被调模块的私有属性？
- [ ] 是否避免直接修改被调模块的内部数据？
- [ ] 是否避免依赖被调模块的实现细节？
- [ ] 是否通过接口修改被调模块的状态？

### 被调模块检查项

- [ ] 是否提供了清晰的公开接口？
- [ ] 是否将内部实现标记为私有（_前缀）？
- [ ] 是否返回数据副本而非内部引用？
- [ ] 是否隐藏了实现细节？
- [ ] 是否提供了状态查询接口？

## 严重程度

### 🔴 高严重度
- 直接修改被调模块的私有属性
- 直接修改被调模块的内部状态
- 绕过接口直接访问内部数据

### 🟡 中严重度
- 访问被调模块的受保护方法
- 依赖被调模块的实现细节
- 重写父类的公开接口

### 🟢 低严重度
- 未使用被调模块提供的便利方法
- 过度依赖被调模块的特定版本

## 主动防护机制

### 1. 开发时实时防护

**在AI生成代码时实时检查：**
- 检测是否正在修改被调模块
- 检测是否直接访问私有属性
- 检测是否绕过公开接口

**实时防护响应：**
```
⚠️ 检测到潜在违规操作！

操作：正在修改 [被调模块名]
风险：可能影响其他调用该模块的组件

请选择：
1. 停止修改，通过创建新模块实现需求
2. 扩展现有模块的公开接口
3. 确认修改并提供充分理由
```

### 2. 修改确认流程

**当AI尝试修改被调模块时，必须回答以下问题：**

#### 问题1：为什么需要修改？
- [ ] 现有接口不满足需求
- [ ] 发现了bug需要修复
- [ ] 需要添加新功能
- [ ] 其他原因：_____

#### 问题2：是否考虑过其他方案？
- [ ] 扩展现有接口？如何扩展：_____
- [ ] 创建新模块？模块名称：_____
- [ ] 通过参数传递实现？
- [ ] 其他方案：_____

#### 问题3：修改影响评估
- [ ] 已确认不影响其他调用者
- [ ] 已评估向后兼容性
- [ ] 已准备更新文档
- [ ] 已准备重新测试

### 3. 模块保护标记

**对于已测试好的模块，添加保护标记：**

```python

# 最后测试时间：2026-03-15
# 测试状态：✅ 通过
# 调用者数量：3个模块
# 修改前请确认影响范围
# ******************************

class 已测试模块:
    """已测试的稳定模块"""
    pass
```

**保护标记触发机制：**
- 检测到保护标记时，强制要求确认流程
- 显示模块的调用者列表
- 要求提供修改理由和影响评估
- 建议优先选择其他方案

### 4. 与模块化开发流程Skill联动

**联动机制：**

1. **模块化开发流程Skill创建调用者时：**
   - 自动触发模块隔离检查器
   - 验证调用方式是否正确
   - 确保没有违规访问

2. **模块隔离检查器发现问题时：**
   - 提示使用模块化开发流程
   - 建议创建新模块或扩展现有接口
   - 提供正确的实现方案

3. **修改被调模块时：**
   - 两个Skill同时触发
   - 模块化开发流程提供替代方案
   - 模块隔离检查器评估修改风险

## 注意事项

1. **严格模式**：本技能采用严格模式，任何违反隔离原则的行为都会被标记
2. **检查范围**：可检查单个模块或整个项目
3. **改进建议**：提供具体的改进建议和示例代码
4. **优先级**：高严重度问题优先处理
5. **主动防护**：在代码生成时实时检查，主动阻止违规操作
6. **修改确认**：修改被调模块时必须经过完整的确认流程
7. **保护标记**：已测试模块添加保护标记，强制确认流程
