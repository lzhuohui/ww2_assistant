# 规范变更审核

## 技能概述

在修改规范文件前进行审核，防止正确规范被改成错误规范。

## 核心原则

1. **保护优先**：默认保护现有规范，变更需充分理由
2. **影响分析**：分析变更对其他规范和代码的影响
3. **用户确认**：重要变更必须向用户确认

## 审核流程

### 步骤1：变更类型识别

| 变更类型 | 风险等级 | 处理方式 |
|----------|----------|----------|
| 新增规范 | 低 | 直接执行 |
| 完善规范（补充说明） | 低 | 直接执行 |
| 修改规范（改变含义） | 高 | 需要审核 |
| 删除规范 | 高 | 需要审核 |

### 步骤2：影响分析

分析变更对以下方面的影响：

1. **其他规范**：是否与其他规范产生矛盾？
2. **现有代码**：是否导致现有代码不合规？
3. **AI执行**：是否影响AI理解和执行？
4. **用户体验**：是否影响用户使用？

### 步骤3：合理性验证

验证变更的合理性：

1. **是否有充分理由？** 用户是否说明了变更原因？
2. **是否符合设计原则？** 是否与规范体系设计原则一致？
3. **是否有更好的方案？** 是否存在不修改规范的替代方案？

### 步骤4：执行决策

| 决策 | 条件 | 动作 |
|------|------|------|
| 直接执行 | 低风险变更 | 执行并记录 |
| 确认后执行 | 中风险变更 | 向用户确认后执行 |
| 建议替代方案 | 高风险变更 | 建议不修改，提供替代方案 |
| 拒绝执行 | 破坏性变更 | 拒绝并说明原因 |

## 审核检查清单

在修改规范前，AI必须检查：

- [ ] 变更类型是什么？（新增/完善/修改/删除）
- [ ] 是否有充分的变更理由？
- [ ] 是否与其他规范产生矛盾？
- [ ] 是否导致现有代码不合规？
- [ ] 是否影响AI理解和执行？
- [ ] 是否有更好的替代方案？

## 常见变更场景处理

### 场景1：用户认为某规范"不合理"

**处理方式**：
1. 分析用户认为不合理的原因
2. 检查是否是理解偏差
3. 如果确实有改进空间，建议完善而非修改
4. 如果是用户特殊需求，建议在特定场景下例外处理

### 场景2：用户要求简化规范

**处理方式**：
1. 分析简化是否会影响规范完整性
2. 如果只是删除冗余内容，可以执行
3. 如果会删除关键约束，建议保留

### 场景3：用户要求添加新规范

**处理方式**：
1. 检查是否与现有规范重复
2. 检查是否与现有规范矛盾
3. 如果合理，直接添加

```
- 变更原因：[用户说明的原因]

- 对其他规范的影响：[无/有，具体说明]

- 对AI执行的影响：[无/有，具体说明]

### 审核结论
- 决策：[直接执行/确认后执行/建议替代方案/拒绝执行]
- 理由：[说明理由]

### 建议
[如有替代方案或改进建议]
```
