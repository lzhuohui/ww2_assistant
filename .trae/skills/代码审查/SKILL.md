# 代码审查器

## 描述
对代码进行全面审查，检查质量、安全性、性能和规范性。

## 使用场景
- 用户要求"审查代码"
- Pull Request合并前检查
- 6A流程中的Approve阶段调用

## 指令

### 步骤1：代码质量检查
- 命名规范（避免"-er"/"-or"后缀）
- 代码结构（函数长度、类复杂度）
- 重复代码识别
- 注释完整性

### 步骤2：安全性检查
- 硬编码密钥/密码
- SQL注入风险
- XSS风险
- 敏感信息泄露

### 步骤3：性能检查
- 循环嵌套深度
- 不必要的计算
- 资源泄漏（文件、连接未关闭）
- 缓存使用建议

### 步骤4：模块边界检查（防止死循环修改）
- 检查是否修改了已冻结模块
- 检查是否修改了非当前任务相关的模块
- 检查是否违反了单向依赖原则
- 检查是否修改了import的模块（被调用模块）

#### 冻结模块检查
- 读取 `.trae/modules/locked/` 目录
- 检查修改的文件是否在锁定列表中
- 如发现违规修改，立即标记为严重错误

#### 依赖关系检查（核心检查）
- 分析当前文件的import语句
- 检查是否修改了被调用模块
- 确保只修改当前模块，不修改依赖模块

#### 被调用模块检查（最重要）
**检查原则：当前模块修改了哪些被它调用的模块**

检查步骤：
1. 读取修改的文件列表
2. 对每个修改的文件，分析其import语句
3. 检查被import的模块是否也被修改
4. 如发现修改了被调用模块，标记为**致命错误**

示例：
```python
# 当前修改了 module_a.py 和 module_b.py

# module_a.py 内容：
from module_b import func_b  # module_a 调用了 module_b
import module_c

# 违规！module_a 调用了 module_b，同时修改了 module_b
# 这将导致 module_b 的其他调用方出错
```

严重级别：
- 🔴 **致命错误**：修改了被调用模块
- 🟠 **严重错误**：修改了已冻结模块
- 🟡 **警告**：修改了非当前任务模块

### 步骤5：生成审查报告
输出 `reports/code_review.md`，包含：
- 问题清单（按严重度分级）
- 模块边界违规（如有，标记为严重）
- 改进建议
- 代码示例（修复前后对比）
