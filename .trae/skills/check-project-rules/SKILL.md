# Project Rules 强制检查技能

## 触发时机

**此技能在每次对话开始时自动触发，无论用户请求什么内容。**

## 执行步骤

### 步骤1：读取 Project Rules

在回应用户之前，必须先读取文件：
```
c:\Users\chmm1\Documents\二战风云\.trae\rules\project-rules.md
```

### 步骤2：建立检查清单

根据project-rules.md的内容，建立以下检查清单：

#### L1 规范检查项
- [ ] 是否遵循二级规范层级（L1 Project总则 > L2 Domain专项规范）
- [ ] 是否检查规则匹配优先级（#Rule > alwaysApply+globs > 手动加载）
- [ ] 是否了解冲突解决策略

#### 编程工作流五阶段检查项
- [ ] **阶段1-需求澄清**：是否已充分理解用户需求，没有歧义
- [ ] **阶段2-收集最佳实践**：是否已收集相关领域的最佳实践
- [ ] **阶段3-提供三方案**：是否已提供三个可行方案供用户选择
- [ ] **阶段4-按规则执行**：是否按照用户选择的方案执行
- [ ] **阶段5-核查交付**：是否在交付前进行了核查

**流转规则检查：**
- [ ] 禁止跳过任何阶段
- [ ] 禁止回退（除非用户主动要求）
- [ ] 阶段3必须等待用户选择方案
- [ ] 阶段1、阶段3、阶段5必须用户确认

#### 模块化设计核心约束检查项
- [ ] 禁止直接修改被调用模块
- [ ] 修改被调用者需评估影响
- [ ] 修改需用户同意

#### 默认值传递规则检查项
- [ ] 默认值来源：被调用模块的默认值由被调用模块自身定义
- [ ] 中间层传递：中间层导入并使用底层模块的默认值，而非硬编码
- [ ] 调用者优先：调用者明确指定值时，优先使用调用者的值

### 步骤3：声明检查完成

在回应用户时，必须首先声明：
```
【Project Rules 检查完成】
- 已读取 project-rules.md
- 已建立检查清单
- 当前任务将遵循五阶段工作流
```

### 步骤4：按规范执行

根据当前任务类型，执行相应的规范：

#### 如果是编程任务
1. 严格遵循五阶段流程
2. 每完成一个阶段，明确告知用户
3. 阶段3必须提供三个方案
4. 等待用户确认后再进入下一阶段

#### 如果是修改/修复任务
1. 评估对其他模块的影响
2. 向用户说明影响范围
3. 获得用户同意后再执行

#### 如果是重构任务
1. 记录原有逻辑形成文档
2. 保留原有版本
3. 新建目录进行重建
4. 确保不缺失原有功能

## 禁止事项

1. **禁止跳过步骤1**：不读取project-rules.md就直接回应用户
2. **禁止跳过检查清单**：不建立检查清单就开始工作
3. **禁止跳过五阶段**：直接执行而不遵循五阶段流程
4. **禁止硬编码默认值**：中间层必须导入底层默认值
5. **禁止直接修改被调用模块**：必须先评估影响并获得用户同意

## 违规后果

如果不执行此技能，将导致：
- 模块化设计违规
- 代码重复和难以维护
- 用户重复提出相同问题
- 项目质量下降
