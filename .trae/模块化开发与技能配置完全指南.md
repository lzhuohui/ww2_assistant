## 一、Trae IDE 技能机制

### 1. 智能体体系

Trae IDE采用**开放的智能体生态**，用户可以：
- 根据需求独立创建智能体
- 分享到市场
- 像插件一样灵活组合智能体

### 2. 技能创建方式

技能不是通过JSON文件配置的，而是通过以下方式创建：

#### 方式一：通过设置中心创建
1. 在IDE模式界面中，点击界面右上角的**设置图标**
2. 在左侧导航栏中，选择**规则和技能**
3. 在**个人规则**部分，点击**+创建**按钮
4. 输入规则内容，点击**保存**

#### 方式二：创建项目规则
1. 在项目根目录创建`.trae/rules`目录
2. 在该目录下创建Markdown文件
3. 添加YAML front matter配置
4. 编写规则内容

### 3. 技能生效方式

| 生效方式 | 说明 | 示例 |
|---------|------|------|
| 始终生效 | 自动在所有对话中应用 | 通用编码规范 |
| 指定文件生效 | 当提及匹配的文件时自动应用 | 提及Python文件时，命名规范自动生效 |
| 智能生效 | AI根据description判断是否应用 | 编写性能敏感代码时，性能优化规范自动生效 |
| 手动触发 | 使用`#Rule`引用规则 | `#Rule 命名规范` |

### 3. 检查规则

#### 规则1：接口隔离原则
**要求**：调用者只能通过公开接口访问被调模块

**✅ 正确示例**：
```python
# 被调模块：数据处理.py
class DataProcessor:
    def process(self, data):
        """公开接口：处理数据"""
        return self._internal_process(data)
    
    def _internal_process(self, data):
        """私有方法：内部实现"""
        return data.strip()

# 调用者：业务逻辑.py
from 数据处理 import DataProcessor
processor = DataProcessor()
result = processor.process(data)  # ✅ 通过公开接口调用
```

**❌ 错误示例**：
```python
# 调用者：业务逻辑.py
from 数据处理 import DataProcessor
processor = DataProcessor()
result = processor._internal_process(data)  # ❌ 直接调用私有方法
```

#### 规则2：数据隔离原则
**要求**：调用者不得直接修改被调模块的内部数据

**✅ 正确示例**：
```python
# 被调模块：配置管理.py
class ConfigManager:
    def __init__(self):
        self._config = {"width": 100}
    
    def get_config(self):
        """公开接口：获取配置副本"""
        return self._config.copy()
    
    def set_config(self, key, value):
        """公开接口：设置配置"""
        self._config[key] = value

# 调用者：界面组件.py
from 配置管理 import ConfigManager
manager = ConfigManager()
config = manager.get_config()  # ✅ 获取副本
manager.set_config("width", 200)  # ✅ 通过接口修改
```

**❌ 错误示例**：
```python
# 调用者：界面组件.py
from 配置管理 import ConfigManager
manager = ConfigManager()
config = manager._config  # ❌ 直接访问内部数据
config["width"] = 200  # ❌ 直接修改内部数据
```

#### 规则3：依赖隔离原则
**要求**：调用者不得依赖被调模块的实现细节

#### 规则4：状态隔离原则
**要求**：调用者不得直接修改被调模块的内部状态

#### 规则5：继承隔离原则
**要求**：子类不得破坏父类的封装性

### 4. 严重程度分级

| 级别 | 类型 | 示例 | 处理方式 |
|------|------|------|----------|
| 🔴 高严重度 | 直接修改私有属性/状态 | `instance._data = new_data` | 拒绝执行，要求重新实现 |
| 🟡 中严重度 | 访问受保护方法/依赖实现细节 | `instance._private_method()` | 提示修正并重新检查 |
| 🟢 低严重度 | 未使用便利方法 | 未使用提供的辅助方法 | 警告并建议修正 |

### 5. 主动防护机制

**1. 开发时实时防护**
- 检测是否正在修改被调模块
- 检测是否直接访问私有属性
- 检测是否绕过公开接口

**2. 修改确认流程**
- 当AI尝试修改被调模块时，强制回答问题
- 为什么修改、替代方案、影响评估

**3. 模块保护标记**
- 对于已测试好的模块，添加保护标记
- 检测到保护标记时，强制要求确认流程

**4. 与模块化开发流程Skill联动**
- 模块化开发流程Skill创建调用者时，自动触发模块隔离检查器
- 模块隔离检查器发现问题时，提示使用模块化开发流程

```
┌─────────────────────────────────────────────────────────────┐
│                    被调模块保护器工作流程                      │
├─────────────────────────────────────────────────────────────┤
│  阶段1：任务开始时 - 建立保护清单                              │
│    - 扫描项目中的所有模块                                      │

└─────────────────────────────────────────────────────────────┘
```

### 4. 保护措施

**措施1：强制确认**
- 任何对已测试模块的修改都必须经过用户确认
- AI不得自行决定修改被调模块
- 默认策略是保护，而非修改

**措施2：替代方案优先**
- 优先建议创建新模块
- 优先建议扩展现有接口
- 优先建议通过参数配置
- 修改被调模块是最后选择

**措施3：影响评估**
- 修改前必须评估影响范围
- 必须识别所有调用者
- 必须评估向后兼容性
- 必须准备测试计划

**措施4：回滚机制**
- 未经同意的修改必须回滚
- 提供一键回滚功能
- 保留修改历史记录

### 5. 保护标记格式

**模块保护标记**：
```python
# *** 已测试模块 - 谨慎修改 ***
# 最后测试时间：2026-03-15
# 测试状态：✅ 通过
# 调用者数量：3个模块
# 修改前请确认影响范围
# ******************************

class 已测试模块:
    """已测试的稳定模块"""
    pass
```
