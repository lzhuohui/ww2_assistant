# ==================== 功能容器设计共识 ====================

"""
## 一、设计理念

### 1.1 核心原则
- 统一性：功能容器作为页面层的容器，统一管理布局和风格
- 复用性：调用通用卡片组件，不重新实现卡片逻辑
- 灵活性：同时支持自定义控件和自动创建卡片
- 一致性：测试架构与实际使用保持一致

### 1.2 功能容器定义
功能容器是组件层的容器组件，包含：
1. 通用容器包装（提供立体风格）
2. 图标显示（对应导航栏功能按钮图标）
3. 标签显示（对应导航栏功能按钮名称）
4. 水平灰色隔断（分隔标题和内容区域）
5. 卡片列表（展示页面内容）

## 二、卡片列表集成方案

### 2.1 问题描述
原始设计中，功能容器只接收已创建好的卡片列表，导致：
- 页面层需要手动创建卡片，代码重复
- 调试逻辑与实际使用不一致
- 测试架构无法真实反映实际使用情况

### 2.2 解决方案
功能容器提供两种方式传入卡片：

**方式一：cards参数（自定义控件）**
```python
FunctionContainer.create(
    config=配置,
    title="系统设置",
    icon="SETTINGS",
    cards=[card1, card2, card3],  # 已创建好的控件
)
```

**方式二：card_configs参数（自动创建通用卡片）**
```python
FunctionContainer.create(
    config=配置,
    title="系统设置",
    icon="SETTINGS",
    card_configs=[
        {"title": "挂机模式", "icon": "POWER_SETTINGS_NEW", "subtitle": "..."},
        {"title": "指令速度", "icon": "SPEED", "subtitle": "..."},
    ],  # 卡片配置列表
)
```

### 2.3 实现原则
- 功能容器内部调用通用卡片的`create()`方法
- 不重新实现卡片创建逻辑
- 复用通用卡片组件，确保UI风格一致

## 三、调试逻辑规范

### 3.1 规范要求
调试逻辑必须真实反映模块的实际使用情况，确保：
- 测试架构与实际使用一致
- 调试逻辑不包含额外的测试代码
- 调试逻辑使用与实际使用相同的参数

### 3.2 实施方案
调试逻辑使用`card_configs`参数，与实际使用方式一致：

```python
if __name__ == "__main__":
    配置 = 界面配置()
    ThemeProvider.initialize(配置)
    
    def main(page: ft.Page):
        page.add(FunctionContainer.create(
            config=配置,
            title="测试功能",
            icon="SETTINGS",
            card_configs=[
                {"title": "测试卡片", "icon": "SETTINGS", "subtitle": "这是一个测试卡片"},
            ],
        ))
    
    ft.run(main)
```

## 四、各层职责

| 层级 | 职责 | 卡片相关 |
|------|------|---------|
| 零件层 | 纯UI，无业务逻辑 | 卡片容器 |
| 组件层 | 组合零件 + 简单交互 | 通用卡片、功能容器 |
| 页面层 | 组装组件 + 业务逻辑 | 传入卡片配置或自定义控件 |

## 五、预期效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 页面层代码量 | 手动创建每个卡片 | 传入配置即可 |
| 测试架构一致性 | 不一致 | 一致 |
| UI风格统一性 | 依赖页面层实现 | 功能容器统一管理 |
| 代码复用性 | 低 | 高 |

## 六、注意事项

1. 功能容器不重新实现卡片逻辑，只调用通用卡片
2. 保持两种传参方式，确保灵活性
3. 调试逻辑必须使用与实际使用相同的参数
4. 页面层可以选择使用card_configs或自定义控件
