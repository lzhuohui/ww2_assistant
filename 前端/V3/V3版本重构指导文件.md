# V3版本重构指导文件

## 一、项目概述

### 1.1 重构目标
- 提升配置文件的人类可维护性
- 实现界面配置与游戏逻辑的解耦
- 建立清晰的扩展机制
- 保持与V2版本的完全隔离

### 1.2 核心改进
| 改进项 | V2现状 | V3方案 |
|--------|--------|--------|
| 配置格式 | JSON（无注释） | YAML（支持注释） |
| 文件粒度 | 单文件1493行 | 每界面一文件~150行 |
| 命名转换 | "设置"↔"界面"字符串替换 | 统一使用界面名称 |
| 游戏对接 | 直接使用界面名称 | 通过ID映射解耦 |
| 扩展机制 | 有限 | 三级扩展（控件/卡片/界面） |

## 三、配置文件规范

### 3.1 界面配置格式（YAML）

```yaml
# ===========================================
# 界面名称配置
# 说明：界面功能描述
# ===========================================

# 界面级布局（全局默认值）
界面布局:
  下拉框宽度: 70
  输入框宽度: 100
  每行控件数: 3
  水平间距: 8
  垂直间距: 6

# -------------------------------------------
# 卡片名称
# 卡片功能描述
# -------------------------------------------
卡片名称:
  卡片信息:
    title: 卡片标题
    icon: ICON_NAME           # Material Icons名称
    subtitle: "卡片副标题"
    enabled: true             # 默认开关状态
    # 卡片级覆盖（可选）
    # dropdown_width: 80
    # controls_per_row: 2
  控件列表:
    - id: 控件ID              # 唯一标识
      type: dropdown          # 控件类型：dropdown/input
      label: 控件标签:        # 标签文本（如需冒号直接加在后面）
      options: ["选项1", "选项2"]  # dropdown专用
      default: 默认值
      # 控件级覆盖（可选）
      # width: 100
      # min_value: 10         # 最小值限制
```

### 3.2 三级扩展机制

```
优先级：控件级 > 卡片级 > 界面级 > 默认值

示例：下拉框宽度
├── 控件.width = 100              → 使用100
├── 卡片.dropdown_width = 80      → 使用80
├── 界面.下拉框宽度 = 70          → 使用70
└── 都没有配置                    → 使用默认值120
```

### 3.3 ID映射表（纯内存生成）

**设计原则**：ID映射表由界面配置动态生成，无需保存文件。

**生命周期**：
```
启动时：扫描界面配置/*.yaml → 生成ID映射 → 存入内存
运行时：使用内存数据
变更时：用户修改界面配置 → 触发刷新ID映射
退出时：清空内存缓存
```

**优点**：
- 数据来源唯一（界面配置）
- 避免文件同步问题
- 简化维护

**内存结构**：
```python
{
  "说明": "界面名称到ID的映射表，由程序自动维护",
  "版本": "1.0",
  "映射": {
    "系统界面.挂机模式.挂机模式": "SYS_001",
    "系统界面.指令速度.指令速度": "SYS_002",
    "账号界面.账号01.类型": "ACC_001"
  },
  "反向映射": {
    "SYS_001": {"界面": "系统界面", "卡片": "挂机模式", "控件": "挂机模式"}
  }
}
```

## 五、层级隔离原则

### 5.1 核心原则

**每个层级只调用下一层级的公开接口，禁止跨层级访问内部实现。**

```
✅ 正确的调用关系：
层级1 → 层级2 → 层级3 → 层级4 → 层级5

❌ 错误的调用关系：
层级3 → 层级5（跳过层级4）
层级2 → 层级5（跳过层级3、4）
```

### 5.2 销毁管理例外

**唯一例外**：层级1直接调用层级5的销毁方法

```python
# 主入口.py（层级1）
def _switch_interface(self, interface_name: str):
    # 销毁旧界面
    old_interface.destroy()
    # 直接调用层级5销毁下拉框数据
    Dropdown.destroy_all_instances()
    # 创建新界面
    ...
```

**原因**：销毁是全局操作，不需要经过中间层级转发。

### 5.3 层级隔离检查清单

| 层级 | 禁止导入 | 禁止调用 |
|------|---------|---------|
| 层级1 | 层级3、4、5内部模块 | 层级3、4、5内部方法（销毁除外） |
| 层级2 | 层级4、5内部模块 | 层级4、5内部方法 |
| 层级3 | 层级5内部模块 | 层级5内部方法 |
| 层级4 | 层级5内部方法 | 层级5私有方法（_开头） |

## 六、下拉框懒加载机制

### 6.1 设计目标

解决大量下拉框导致界面卡顿问题：
- 进入界面时：只加载当前值，不加载选项列表
- 点击下拉框时：才加载对应选项列表
- 点击另一个下拉框时：销毁上一个下拉框的选项列表
- 切换界面时：销毁所有下拉框数据

### 6.2 懒加载逻辑流程

| 场景 | 行为 | 说明 |
|------|------|------|
| 创建下拉框 | 不加载选项 | state["options_loaded"] = False |
| 第一次点击 | 加载选项 | 调用get_options() |
| 再次点击同一个 | 跳过加载 | 已加载，避免重复 |
| 点击另一个 | 销毁上一个，加载当前 | _close_all_dropdowns() |
| 切换界面 | 销毁所有 | destroy_all_instances() |

### 6.3 销毁内容说明

| 销毁内容 | 说明 | 时机 |
|---------|------|------|
| options | 选项列表数据（内存） | 点击另一个下拉框 / 切换界面 |
| options_loaded | 是否已加载标志 | 点击另一个下拉框 / 切换界面 |
| _all_dropdowns | 类变量，存储所有实例引用 | 切换界面 |

**不销毁的内容**：
- 控件本身（Flet框架管理）
- 配置数据（持久化存储）

### 6.4 关键方法

| 方法 | 说明 | 调用者 |
|------|------|--------|
| get_raw_value() | 获取原始配置值，不加载选项列表 | 下拉框内部 |
| get_options() | 获取选项列表，仅在点击时调用 | 下拉框内部 |
| _close_all_dropdowns() | 销毁上一个下拉框的选项 | 下拉框内部 |
| destroy_all() | 销毁当前实例的所有下拉框数据 | 下拉框内部 |
| destroy_all_instances() | 销毁所有实例的选项数据 | 层级1调用 |

### 6.5 模块化设计

**调用者职责**：
- 只传递定位信息：interface, card, control_id
- 不需要知道懒加载细节
- 不需要管理销毁（由层级1负责）

**下拉框模块职责**：
- 自己获取数据（get_raw_value/get_options）
- 内部管理懒加载状态
- 内部管理销毁逻辑
- 提供类方法供层级1调用

### 6.6 性能优化效果

| 对比项 | 优化前 | 优化后 |
|--------|--------|--------|
| 进入建筑界面 | 加载几十个下拉框的所有选项 | 只加载当前值 |
| 界面响应 | 卡顿明显 | 流畅 |
| 内存占用 | 选项积累 | 点击时加载，切换时销毁 |

## 七、层级5基础模块规范

### 7.1 模块分类

| 类型 | 模块 | 特征 | 接口需求 |
|------|------|------|---------|
| **数据交互型** | 下拉框、输入框 | 需要读取/保存配置 | get_value/set_value |
| **编辑交互型** | 头像 | 需要用户编辑 | set_text/get_text |
| **纯显示型** | 图标、标签、分割线 | 静态展示 | 无需额外接口 |

### 7.2 统一架构模式

所有模块遵循相同模式：

```python
class ModuleName:
    _config_manager: ConfigManager = None
    
    @classmethod
    def set_config_manager(cls, config_manager: ConfigManager):
        """设置配置管理实例"""
        cls._config_manager = config_manager
    
    @staticmethod
    def _check_config_manager():
        """检查配置管理是否已设置"""
        if ModuleName._config_manager is None:
            raise RuntimeError("模块未设置config_manager")
    
    @staticmethod
    def get_xxx_size() -> int:
        """获取配置值，带后备值"""
        ModuleName._check_config_manager()
        value = ModuleName._config_manager.get_ui_size(...)
        if value is None:
            value = 默认值
        return value
```

### 7.3 后备值规范

后备值必须与配置管理的深色主题默认值一致：

| 属性 | 正确后备值 | 错误示例 |
|------|-----------|---------|
| text_primary | `#FFFFFF` | - |
| text_secondary | `#B0B0B0` | ~~`#C5C5C5`~~ |
| text_disabled | `#666666` | ~~`#656565`~~ |
| bg_card | `#2D2D2D` | ~~`#2D2D4A`~~ |
| border | `#3D3D3D` | ~~`#3D3D5C`~~ |
| accent | `#0078D4` | - |
| bg_primary | `#202020` | ~~`#1A1A2E`~~ |

2. **禁止冗余导入**：移除未使用的导入

## 十二、接口设计规范

### 12.1 是否公开接口的判断标准

| 条件 | 公开接口 | 不公开接口 |
|------|---------|-----------|
| 数据持久化 | ✅ 需要 | ❌ 不需要 |
| 动态更新频繁 | ✅ 需要 | ❌ 不需要 |
| 纯显示组件 | ❌ 不需要 | ✅ 不需要 |
| Flet原生支持 | ❌ 不需要 | ✅ 不需要 |

### 12.2 数据持久化责任

```
┌─────────────────────────────────────────────────────────────────┐
│                         调用者层面                               │
│  - 启动时读取保存的值                                            │
│  - 创建组件时传入值                                              │
│  - 用户修改时保存值                                              │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      基础/复合模块                               │
│  职责：                                                          │
│  - 显示传入的数据                                                │
│  - 提供编辑功能（如需要）                                        │
│  - 通过回调通知数据变化                                          │
│                                                                 │
│  不负责：                                                        │
│  - 数据持久化 ← 这是调用者的责任                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 12.3 示例：头像模块

```python
# 调用者代码
def create_avatar():
    # 1. 启动时读取保存的值
    avatar_text = config_manager.get_user_info("avatar_text", "帅")
    
    # 2. 创建头像，传入保存的值
    avatar = Avatar.create(
        text=avatar_text,
        on_text_change=lambda new_text: save_avatar_text(new_text)
    )

def save_avatar_text(new_text):
    # 3. 用户修改时保存
    config_manager.set_user_info("avatar_text", new_text)
```
