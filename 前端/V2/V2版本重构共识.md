## 二、当前问题

### 2.1 原代码问题

| 问题类型 | 具体问题 | 影响 |
|---------|---------|------|
| 配置结构不一致 | 用户配置.json扁平结构 vs game_config.json按界面分类 | 数据转换复杂 |
| 选项列表难维护 | 按类型分类（数字/文本），不知道对应哪个界面 | 维护困难 |
| 建筑等级统一化 | 所有建筑用相同等级范围，实际每个建筑不同 | 配置错误 |
| 模块耦合度高 | 调用者需要了解被调模块内部参数和行为 | 难以维护 |
| 界面与游戏控制分离 | 界面设置与游戏控制逻辑使用不同配置结构 | 系统割裂 |

### 1.2 重构目标

1. **统一配置结构**：界面设置与游戏控制使用相同的配置结构
2. **完全模块化**：模块公开接口，隐藏内部逻辑
3. **按界面分类**：配置文件按界面分类，方便维护
4. **懒加载优化**：下拉框采用懒加载，减少内存占用

### 2.2 层级职责详细说明

#### 层级1：主入口V2

**职责**：
- 初始化服务
- 创建导航面板（左侧）
- 导航切换时调用功能界面模块
- 页面销毁时调用销毁方法

**不负责**：
- 功能界面内部逻辑
- 控件创建和管理

**内容**：
```
主入口V2
├── 导航面板（左侧）
│   └── 导航按钮 × N
└── 内容区域（右侧）
    └── 功能界面V2
```

#### 层级2：功能界面V2

**职责**：
- 调用卡片组
- 布局卡片

**不负责**：
- 创建控件
- 创建开关
- 销毁逻辑（由卡片组负责）

**内容**：
```
功能界面V2（如建筑界面V2）
├── 卡片组V2 × N
└── 布局（Column）
```

#### 层级3：卡片组V2

**职责**：
- 创建开关
- 创建控件（下拉框、输入框）
- 布局控件
- 销毁下拉框菜单

**不负责**：
- 导航逻辑
- 方案切换

**内容**：
```
卡片组V2
├── 卡片（左侧开关+右侧控件区）
│   ├── 左侧：开关
│   └── 右侧：控件区
│       └── 控件 × N（下拉框/输入框）
└── 内部管理：创建控件、布局控件、销毁控件
```

#### 层级4：复合模块

| 模块 | 组成 | 职责 | 销毁 |
|------|------|------|------|
| 图标标题 | 图标+标题+分割线 | 左侧标题区 | 无 |
| 导航按钮 | 卡片+通用按钮 | 导航按钮 | 无 |
| 卡片 | 左侧开关+右侧控件区 | 卡片容器 | 无 |

#### 层级5：基础模块

| 模块 | 职责 | 自管理 | 销毁 |
|------|------|--------|------|
| 卡片容器 | 布局容器 | 无 | 无 |
| 输入框 | 文本输入+配置保存 | 无 | 无 |
| 下拉框 | 懒加载+销毁+配置保存 | 销毁菜单 | **有** |
| 主题色块 | 主题颜色块 | 无 | 无 |

## 三、销毁逻辑

### 3.1 销毁原则

```
┌─────────────────────────────────────────────────────────────┐
│                    销毁原则                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 谁创建，谁销毁                                          │
│     • 下拉框菜单由下拉框创建 → 下拉框销毁                    │
│     • 卡片组由功能界面创建 → 功能界面调用卡片组销毁          │
│     • 功能界面由主入口创建 → 主入口调用功能界面销毁          │
│                                                             │
│  2. 切换时销毁上一界面                                      │
│     • 防止内存积累                                          │
│     • 防止控件残留                                          │
│                                                             │
│  3. 只销毁需要销毁的                                        │
│     • 下拉框：需要销毁（懒加载菜单）                         │
│     • 输入框：不需要销毁                                    │
│     • 开关：不需要销毁                                      │
│     • 卡片容器：不需要销毁                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 销毁链路

```
┌─────────────────────────────────────────────────────────────┐
│                    销毁链路                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  触发时机：导航按钮切换界面时                                │
│                                                             │
│  主入口V2._switch_page(page_name)                           │
│      │                                                      │
│      ├── 1. 销毁上一界面                                    │
│      │       ↓                                              │
│      │   功能界面V2.destroy()                               │
│      │       ↓                                              │
│      │   卡片组V2.dropdown.destroy_all()                    │
│      │       ↓                                              │
│      │   下拉框V2.destroy_all()  ← 下拉框负责销毁菜单       │
│      │                                                      │
│      └── 2. 创建新界面                                      │
│              ↓                                              │
│          功能界面V2.create()                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**注意**：卡片组V2不需要destroy_all()方法，因为：
- 下拉框V2自己管理销毁
- 输入框V2不需要销毁
- 开关不需要销毁
- 卡片容器不需要销毁

### 3.3 销毁责任总结

| 层级 | 模块 | 是否需要销毁方法 | 说明 |
|------|------|-----------------|------|
| 1 | 主入口V2 | 需要 | 调用功能界面.destroy() |
| 2 | 功能界面V2 | 需要 | 调用卡片组.dropdown.destroy_all() |
| 3 | 卡片组V2 | **不需要** | 提供dropdown属性访问 |
| 5 | 下拉框V2 | **需要** | 销毁懒加载菜单 |
| 5 | 输入框V2 | 不需要 | 没有需要销毁的东西 |

## 五、模块化设计原则

### 5.1 核心原则

**调用者只能做三件事**：
1. 调用模块
2. 布局逻辑
3. 数据回调逻辑

**禁止**：在调用者中包含任何影响被调模块内部行为的逻辑

**数据回调逻辑说明**：

```
┌─────────────────────────────────────────────────────────────┐
│                    数据回调逻辑                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  什么是数据回调？                                            │
│  ───────────────                                            │
│  当用户操作控件时触发的数据处理函数                          │
│                                                             │
│  数据回调只处理数据，不处理UI：                              │
│  ────────────────────────                                   │
│  ✓ 日志记录                                                 │
│  ✓ 数据验证                                                 │
│  ✓ 通知其他模块                                             │
│  ✗ 更新显示（由被调模块处理）                                │
│  ✗ 改变颜色（由被调模块处理）                                │
│  ✗ 显示提示（由被调模块处理）                                │
│                                                             │
│  V1 vs V2 对比：                                            │
│  ─────────────                                              │
│  V1：回调中包含UI处理逻辑（错误）                            │
│  V2：回调只处理数据，UI由被调模块自己管理（正确）            │
│                                                             │
│  示例：                                                     │
│  ────────                                                   │
│  # V1错误做法                                               │
│  def on_change(value):                                      │
│      dropdown.set_value(value)  # UI处理，不应该在这里      │
│      dropdown.update()          # UI处理，不应该在这里      │
│                                                             │
│  # V2正确做法                                               │
│  def on_change(section, control_id, value):                 │
│      # 只处理数据，不处理UI                                 │
│      logger.log(f"配置变更: {section}.{control_id}")        │
│      # UI由下拉框V2自己管理                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 模块隔离设计原则

**在整个重构过程中，所有模块必须功能独立，互不交叉**

```
┌─────────────────────────────────────────────────────────────┐
│                    模块隔离设计原则                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 单一职责原则                                            │
│  ─────────────                                              │
│  • 每个模块只负责一个功能                                   │
│  • 不要在一个模块中实现多个不相关的功能                     │
│                                                             │
│  2. 模块隔离原则                                            │
│  ─────────────                                              │
│  • 修改一个模块不影响其他模块                               │
│  • 模块之间通过公开接口通信                                 │
│  • 内部逻辑不暴露给其他模块                                 │
│                                                             │
│  3. 影响范围控制                                            │
│  ─────────────                                              │
│  • 修改卡片开关 → 只影响卡片开关模块                        │
│  • 修改卡片控件 → 只影响卡片控件模块                        │
│  • 修改下拉框 → 只影响下拉框模块                            │
│  • 修改输入框 → 只影响输入框模块                            │
│                                                             │
│  4. V1版本问题回顾                                          │
│  ─────────────                                              │
│  • SwitchableCard职责过多：布局+开关+控件创建+控件管理      │
│  • 修改一个界面导致大面积故障                               │
│  • 几个小时无法恢复                                         │
│  • 根本原因：模块功能交叉                                   │
```

```python
        """提供下拉框实例访问（用于销毁）"""
        return self._card_controls.dropdown
    
    def create(
        self,
        section: str,
        title: str,
        icon: str,
        controls_config: List[Dict],
        has_switch: bool = True,
        on_change: Callable = None,

        # 1. 创建开关（左侧）

        controls_area = self._card_controls.create(section, controls_config, on_change)
        
        # 3. 布局（左侧 + 右侧）
        return ft.Row([
            switch_control,    # 左侧：开关
            controls_area,     # 右侧：控件区
        ])
```

## 十五、重构教训（重要）

### 15.1 重构失败的典型错误

在重构过程中，AI犯了以下严重错误，导致重构2-3小时无进展：

| 错误类型 | 具体表现 | 后果 |
|----------|----------|------|
| **不知道重构基础** | 不知道重构是基于`前端\游戏设置界面` | 完全偏离方向 |
| **忽略原布局** | 没有认真阅读原主入口布局 | 创建了错误的布局结构 |
| **忽略用户信息卡片** | 只看到左侧导航+右侧内容，漏掉用户信息卡片 | 布局不完整 |
| **忽略导航按钮组** | 不知道有11个导航项 | 功能不完整 |
| **忽略懒加载逻辑** | 没有实现点击加载、关闭销毁 | 核心功能缺失 |
| **层级职责混淆** | 层级4承担了层级3的职责 | 架构混乱 |

## 十六、卡片加载机制（重要）

### 16.1 概念区分

| 概念 | 说明 | 示例 |
|------|------|------|
| **控件值** | 卡片显示的数据值（副标题、下拉框选中值、输入框文本值） | "全自动" "100" "15" |
| **控件列表** | 卡片展开后的交互控件（下拉框、输入框等） | Dropdown, TextField |

### 16.2 加载流程

```
┌─────────────────────────────────────────────────────────────┐
│                    卡片加载流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 启动主入口                                               │
│     ────────────                                            │
│     • 进入系统界面                                           │
│                                                             │
│  2. 进入功能界面                                             │
│     ────────────                                            │
│     • 显示全部展开的完整界面                                 │
│     • 加载所有卡片的控件值（从配置读取）                     │
│     • 不加载控件列表                                         │
│                                                             │
│  3. 加载第一个卡片的控件列表                                 │
│     ────────────────────────                                │
│     • 设置卡片加载标识 = True                                │
│     • 记录 current_loaded_card                               │
│                                                             │
│  4. 切换卡片                                                 │
│     ────────────                                            │
│     • 检查当前卡片是否有加载标识                             │
│     • 有加载标识 → 销毁控件列表 → 加载标识 = False           │
│     • 加载新卡片的控件列表 → 加载标识 = True                 │
│     • 更新 current_loaded_card                               │
│                                                             │
│  5. 已加载的卡片不重复加载                                   │
│     ─────────────────────                                   │
│     • 检查加载标识                                           │
│     • 已加载则跳过                                           │
│                                                             │
│  6. 切换界面                                                 │
│     ────────────                                            │
│     • 销毁原界面所有已加载的控件列表                         │
│     • 进入新界面 → 重复步骤2-5                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 16.3 关键设计

| 设计要点 | 说明 |
|----------|------|
| 控件值 vs 控件列表 | 控件值是显示数据，控件列表是交互控件 |
| 加载标识 | 每个卡片有 `loaded` 标识，避免重复加载 |
| 销毁时机 | 切换卡片时销毁旧卡片控件，切换界面时销毁所有控件 |
| 内存优化 | 始终只有一个卡片的控件列表在内存中 |

### 16.4 PopupMenuButton 的懒加载特性

**重要发现**：PopupMenuButton 本身就实现了懒加载机制！

| 时机 | 行为 |
|------|------|
| 创建时 | 只存储选项列表数据（内存占用小） |
| 点击打开时 | 才渲染菜单项到DOM（真正的性能消耗） |
| 关闭时 | 菜单项仍在内存中 |
| 导航切换时 | 销毁所有卡片，释放内存 |

**结论**：PopupMenuButton 的内置懒加载机制天然满足了性能需求，无需额外实现复杂的显式懒加载逻辑。

### 16.5 避免的错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 加载所有卡片的控件列表 | 内存积累，退出卡顿 | 只加载当前卡片 |
| 不销毁旧卡片控件 | 内存泄漏 | 切换卡片时销毁 |
| 重复加载已加载卡片 | 性能浪费 | 检查加载标识 |
| 过度设计复杂逻辑 | 代码混乱，难以维护 | 简单直接，按流程执行 |
| 死板执行共识要求 | 代码复杂，反而不流畅 | 灵活调整，目标是流畅操作 |

### 16.6 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                    核心原则                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 共识要求只是参考思路                                    │
│     ─────────────────                                      │
│     • 最终目标是流畅操作                                    │
│     • 具体实现可以灵活调整                                  │
│     • 不必死板执行字面要求                                  │
│                                                             │
│  2. 简单直接优于复杂设计                                    │
│     ─────────────────                                      │
│     • PopupMenuButton 天然满足性能需求                      │
│     • 无需额外实现复杂的显式懒加载逻辑                      │
│                                                             │
│  3. 保持原有布局不变                                        │
│     ─────────────────                                      │
```

```python
        name = config_service.get_value(section, "名称")
        
        if account_type == "主帅" and is_enabled and name:
            commanders.append(name)
    return commanders
```

### 19.2 联动逻辑

| 控件 | 选项来源 |
|------|----------|
| 主要统帅 | 所有主帅列表 |
| 次要统帅 | 所有主帅列表 - 已选主要统帅 |

### 19.3 设计原因

- 不使用输入框：避免输入错误导致游戏控制逻辑失败
- 从账号配置获取：确保数据可靠，避免手动输入错误

## 二十、配置方案管理设计

### 20.1 卡片布局

```
┌─────────────────────────────────────────────────────────────┐
│ [图标] 01方案                                               │
│        ───────────────────────────────────────────────────  │
│        [加载配置]  [输入框: 高速挂机]  [保存配置]            │
│                       ↑                                     │
│            提示："输入名称保存当前配置"                      │
│            已保存则显示上次名称（最多4个字）                 │
└─────────────────────────────────────────────────────────────┘
```

### 20.2 按钮文字替换提示

| 按钮 | 正常状态 | 点击反馈 |
|------|----------|----------|
| **加载配置** | "加载配置" | "请输入名称" / "配置不存在" / "加载成功" |
| **保存配置** | "保存配置" | "请输入名称" / "保存成功" |

**反馈机制**：按钮文字即时显示操作结果，1.5秒后恢复原文字

### 20.3 输入框限制

- 最多输入4个字
- 不显示"0/4"计数器
- 通过 `on_change` 手动截断

### 20.4 账号界面联动

每个账号卡片增加"配置方案"下拉框：
- 默认选项："默认配置"
- 其他选项：用户保存的配置名称

### 20.5 操作流程

```
1. 用户在各界面设置数据
2. 进入配置方案界面
3. 在某个方案卡片输入框输入名称（最多4个字）
4. 点击"保存配置" → 保存当前所有配置数据
5. 进入账号界面
6. 每个账号下拉框选择配置方案
7. 游戏控制按选定方案执行
```
