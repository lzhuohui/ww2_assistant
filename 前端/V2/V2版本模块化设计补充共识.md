**版本**: 1.0  
**日期**: 2026-03-30  
**状态**: 已确认

## 一、背景

在V2重构过程中，发现了以下违反模块化设计原则的问题：

| 问题类型 | 具体表现 | 违反原则 |
|---------|---------|---------|
| 职责混淆 | 卡片组承担了开关创建、控件创建、布局等多个职责 | 单一职责原则 |
| 数据归属错误 | 主入口传递用户信息卡片的具体数据字段 | 模块隔离原则 |
| 调用者越权 | 调用者包含被调模块内部数据逻辑 | 核心原则 |

## 二、核心设计规范

### 2.1 数据归属原则

```
┌─────────────────────────────────────────────────────────────┐
│                    数据归属原则                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  规则1：数据由使用模块自己获取                               │
│  ─────────────────────────────                              │
│  ✓ 模块从配置服务获取数据                                   │
│  ✗ 调用者传递具体数据字段                                   │
│                                                             │
│  规则2：调用者只传递服务引用                                 │
│  ─────────────────────────────                              │
│  ✓ 传递 config_service 对象                                │
│  ✗ 传递 username="xxx", count=5 等具体值                   │
│                                                             │
│  规则3：模块内部定义默认值                                   │
│  ─────────────────────────────                              │
│  ✓ 模块内部定义 DEFAULT_USERNAME = "默认用户"              │
│  ✗ 调用者提供默认值                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 正确示例对比

**错误做法（V1风格）**：
```python
# 主入口.py - 调用者传递具体数据
def _create_user_card(self):
    return UserCard.create(
        username="二战风云玩家",      # ✗ 调用者传递数据
        authorized_count=0,           # ✗ 调用者传递数据
        max_accounts=15,              # ✗ 调用者传递数据
        expire_days=30,               # ✗ 调用者传递数据
        theme_colors=self._theme_colors,
    )
```

**正确做法（V2风格）**：
```python
# 主入口.py - 调用者只传递服务引用
def _create_user_card(self):
    user_card = UserCard(self._config_service)  # ✓ 传递服务引用
    return user_card.create(theme_colors=self._theme_colors)

# 用户信息卡片.py - 模块自己获取数据
class UserCard:
    DEFAULT_USERNAME = "二战风云玩家"  # ✓ 模块定义默认值
    
    def __init__(self, config_service):
        self._config_service = config_service
    
    def create(self, theme_colors):
        username = self._get_username()  # ✓ 模块自己获取
        ...
    
    def _get_username(self):
        if self._config_service:
            return self._config_service.get_user_preference("username", DEFAULT_USERNAME)
        return DEFAULT_USERNAME
```

## 三、模块职责拆分规范

### 3.1 拆分原则

```
┌─────────────────────────────────────────────────────────────┐
│                    模块职责拆分原则                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  规则1：一个模块只负责一件事                                 │
│  ───────────────────────                                    │
│  ✓ 卡片开关：只负责开关逻辑                                  │
│  ✓ 卡片控件：只负责控件创建                                  │
│  ✓ 卡片组：只负责调用和布局                                  │
│                                                             │
│  规则2：模块间通过公开接口通信                               │
│  ───────────────────────                                    │
│  ✓ 卡片组调用卡片开关.create()                              │
│  ✓ 卡片组调用卡片控件.create()                              │
│  ✗ 卡片组直接创建开关控件                                    │
│  ✗ 卡片组直接创建下拉框控件                                  │
│                                                             │
│  规则3：修改影响范围控制                                     │
│  ───────────────────────                                    │
│  ✓ 修改开关逻辑 → 只影响卡片开关模块                        │
│  ✓ 修改控件创建 → 只影响卡片控件模块                        │
│  ✗ 修改卡片组 → 影响所有子模块                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 卡片组拆分示例

**错误做法（职责混淆）**：
```python
# 卡片组.py - 承担了太多职责
class CardGroup:
    def create(self, section, title, icon, controls_config):
        # ✗ 创建开关
        switch = ft.Switch(...)
        
        # ✗ 创建下拉框
        dropdown = self._dropdown.create(...)
        
        # ✗ 创建输入框
        input_box = self._input_box.create(...)
        
        # ✓ 布局（这是正确的）
        return ft.Row([switch, dropdown, input_box])
```

**正确做法（职责分离）**：
```python
# 卡片开关.py - 只负责开关
class CardSwitch:
    @staticmethod
    def create(title, icon, enabled, on_toggle):
        # ✓ 只创建开关相关内容
        return ft.Container(...)

# 卡片控件.py - 只负责控件
class CardControls:
    def create(self, section, controls_config, enabled, on_change):
        # ✓ 只创建控件相关内容
        return ft.Container(...)

# 卡片组.py - 只负责调用和布局
class CardGroup:
    def __init__(self, page, config_service):
        self._card_controls = CardControls(page, config_service)
    
    def create(self, section, title, icon, controls_config):
        # ✓ 调用卡片开关
        card_switch = CardSwitch.create(title, icon, enabled, on_toggle)
        
        # ✓ 调用卡片控件
        controls_area = self._card_controls.create(section, controls_config, enabled, on_change)
        
        # ✓ 布局
        return ft.Row([card_switch, controls_area])
```

## 四、检查清单

### 4.1 调用者检查清单

在编写调用代码时，检查以下项目：

- [ ] 是否只传递了服务引用（如 config_service）？
- [ ] 是否避免了传递具体数据字段？
- [ ] 是否只做了三件事：调用模块、布局逻辑、数据回调逻辑？
- [ ] 是否没有包含被调模块内部行为的逻辑？

### 4.2 被调模块检查清单

在编写模块代码时，检查以下项目：

- [ ] 是否定义了默认值常量？
- [ ] 是否从配置服务获取数据？
- [ ] 是否只负责一件事？
- [ ] 是否通过公开接口暴露功能？
- [ ] 是否没有包含被调模块内部行为的逻辑？
- [ ] 请全面核查完成的模块是否和这规范总结有冲突,特别是模块独立,调用和被调用之间的交叉重复

## 五、常见错误案例记录

### 5.1 数据获取职责错位（2026-03-30修正）

**错误表现**：调用者代替被调模块获取数据

```python
# 卡片控件.py - 错误做法
class CardControls:
    def _create_dropdown_control(self, section, control_id, ...):
        # ✗ 调用者获取选项列表
        options = self._config_service.get_options(section, control_id)
        # ✗ 调用者获取当前值
        current_value = self._config_service.get_text_value(section, control_id)
        
        # 传递给下拉框
        dropdown_container = self._dropdown.create(
            options=options,           # ✗ 传递具体数据
            current_value=current_value,  # ✗ 传递具体数据
        )
```

**正确做法**：被调模块自己获取数据

```python
# 卡片控件.py - 正确做法
class CardControls:
    def _create_dropdown_control(self, section, control_id, ...):
        # ✓ 只传递section和control_id
        dropdown_container = self._dropdown.create(
            section=section,           # ✓ 传递定位信息
            control_id=control_id,     # ✓ 传递定位信息
        )

# 下拉框.py - 模块自己获取数据
class Dropdown:
    def create(self, section, control_id, ...):
        # ✓ 模块自己获取选项列表
        options = self._get_options(section, control_id)
        # ✓ 模块自己获取当前值
        current_value = self._get_current_value(section, control_id, options)
    
    def _get_options(self, section, control_id):
        if self._config_service:
            return self._config_service.get_options(section, control_id)
        return DEFAULT_OPTIONS
    
    def _get_current_value(self, section, control_id, options):
        if self._config_service:
            return self._config_service.get_text_value(section, control_id)
        return options[0] if options else {"text": "", "value": ""}
```

### 5.2 开关状态获取重复（2026-03-30修正）

**错误表现**：调用者和被调模块都获取开关状态

```python
# 卡片组.py - 错误做法
class CardGroup:
    def create(self, section, ...):
        # ✗ 调用者获取开关状态
        enabled = self._config_service.get_enabled(section)
        
        # ✗ 传递给卡片开关
        card_switch = CardSwitch.create(
            enabled=enabled,  # ✗ 传递具体数据
        )
        
        # ✗ 又传递给卡片控件
        controls_container = self._card_controls.create(
            enabled=enabled,  # ✗ 重复传递
        )
```

**正确做法**：被调模块自己获取开关状态

```python
# 卡片组.py - 正确做法
class CardGroup:
    def create(self, section, ...):
        # ✓ 只传递section，不获取enabled
        card_switch = self._card_switch.create(
            section=section,  # ✓ 传递定位信息
        )
        
        # ✓ 从卡片开关获取状态（用于布局）
        enabled = card_switch.get_state()
        
        # ✓ 卡片控件自己获取enabled
        controls_container = self._card_controls.create(
            section=section,  # ✓ 不传递enabled
        )

# 卡片开关.py - 模块自己获取状态
class CardSwitch:
    def create(self, section, ...):
        # ✓ 模块自己获取开关状态
        enabled = self._get_enabled(section)
    
    def _get_enabled(self, section):
        if self._config_service:
            return self._config_service.get_enabled(section, True)
        return True

# 卡片控件.py - 模块自己获取状态
class CardControls:
    def create(self, section, ...):
        # ✓ 模块自己获取开关状态
        enabled = self._get_enabled(section)
    
    def _get_enabled(self, section):
        if self._config_service:
            return self._config_service.get_enabled(section, True)
        return True
```

### 5.3 数据保存职责错位（2026-03-30修正）

**错误表现**：调用者代替被调模块保存数据

```python
# 卡片控件.py - 错误做法
class CardControls:
    def _create_dropdown_control(self, section, control_id, ...):
        def on_change(value):
            # ✗ 调用者保存数据
            self._config_service.set_value(section, control_id, value)
        
        dropdown_container = self._dropdown.create(
            on_change=on_change,  # ✗ 调用者处理保存
        )
```

**正确做法**：被调模块自己保存数据

```python
# 下拉框.py - 正确做法
class Dropdown:
    def create(self, section, control_id, ...):
        def handle_select(value):
            # ✓ 模块自己保存数据
            self._save_value(section, control_id, value)
            
            # 通知调用者（可选）
            if on_change:
                on_change(section, control_id, value)
    
    def _save_value(self, section, control_id, value):
        if self._config_service:
            self._config_service.set_value(section, control_id, value)
```

### 5.4 错误总结

| 错误类型 | 错误表现 | 正确做法 |
|---------|---------|---------|
| 数据获取错位 | 调用者获取数据后传递给被调模块 | 被调模块自己获取数据 |
| 数据保存错位 | 调用者处理被调模块的数据保存 | 被调模块自己保存数据 |
| 状态获取重复 | 调用者和被调模块都获取同一状态 | 被调模块自己获取，调用者通过接口获取 |
| 参数传递冗余 | 传递具体数据值 | 只传递定位信息（section、control_id） |

## 六、尺寸传递链规范

### 6.1 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                    尺寸传递链原则                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 一层一层传递：不跨级传递尺寸                             │
│  2. 被调模块定义：基础尺寸由被调模块定义                     │
│  3. 调用者继承：调用者通过公开接口从被调模块获取             │
│  4. 特殊覆盖：调用者有特殊需求时可定义自己的尺寸覆盖         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 尺寸传递链示例

```
层级5: 卡片容器.py
─────────────────
定义: DEFAULT_HEIGHT = 100, DEFAULT_PADDING = 6
暴露: get_height(), get_padding(), get_width()...

层级4: 卡片开关.py / 卡片.py
─────────────────
继承: CardContainer.get_height(), CardContainer.get_padding()
定义: DEFAULT_ICON_SIZE, DEFAULT_TITLE_SIZE...
暴露: get_height(), get_icon_size(), get_title_size()...

层级3: 卡片组.py
─────────────────
继承: CardSwitch.get_height(), CardSwitch.get_padding()
暴露: get_card_height(), get_padding()

层级2: 功能界面
─────────────────
继承: CardGroup.get_card_height()
```

### 6.3 代码示例

**被调模块（层级5）定义并暴露**：
```python
# 卡片容器.py
class CardContainer:
    DEFAULT_HEIGHT = 100
    DEFAULT_PADDING = 6
    
    @staticmethod
    def get_height():
        return CardContainer.DEFAULT_HEIGHT
    
    @staticmethod
    def get_padding():
        return CardContainer.DEFAULT_PADDING
```

**调用者（层级4）继承并扩展**：
```python
# 卡片开关.py
from 前端.V2.层级5_基础模块.卡片容器 import CardContainer

class CardSwitch:
    DEFAULT_ICON_SIZE = 22  # 本模块定义的尺寸
    
    @staticmethod
    def get_height():
        return CardContainer.get_height()  # 从被调模块继承
    
    @staticmethod
    def get_icon_size():
        return CardSwitch.DEFAULT_ICON_SIZE  # 本模块定义
```

**调用者（层级3）继续继承**：
```python
# 卡片组.py
from 前端.V2.层级4_复合模块.卡片开关 import CardSwitch

class CardGroup:
    @staticmethod
    def get_card_height():
        return CardSwitch.get_height()  # 从被调模块继承
```

## 七、主题配置管理规范

### 7.1 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                    主题配置管理原则                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 集中存储：主题配置统一存储在用户偏好.json               │
│  ────────────────────────────────────────                   │
│  ✓ 配置文件：前端/配置/V2/用户偏好.json                     │
│  ✗ 分散定义：各模块中硬编码颜色值                           │
│                                                             │
│  2. 统一接口：通过配置服务获取主题配置                      │
│  ────────────────────────────────────────                   │
│  ✓ config_service.get_theme_colors()                       │
│  ✓ config_service.get_current_theme()                      │
│  ✓ config_service.set_current_theme(theme_name)            │
│  ✓ config_service.get_accent_color()                       │
│                                                             │
│  3. 基础模块fallback：层级5模块可定义DEFAULT_THEME作为默认  │
│  ────────────────────────────────────────                   │
│  ✓ 确保模块可独立运行测试                                   │
│  ✓ 上层模块通过theme_colors参数传递实际主题                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 主题配置数据流

```
用户偏好.json
     │
     ▼
配置仓库.py (DEFAULT_THEME_COLORS作为fallback)
     │
     ▼
配置服务.py (get_theme_colors, set_current_theme等)
     │
     ▼
主入口.py (调用config_service.get_theme_colors())
     │
     ▼
功能界面 (通过theme_colors参数接收)
     │
     ▼
基础模块 (使用theme_colors或DEFAULT_THEME)
```

### 7.3 正确示例

**配置文件（用户偏好.json）**：
```json
{
  "主题": {
    "当前主题": "dark",
    "当前强调色": "blue"
  },
  "主题配置": {
    "light": {...},
    "dark": {...}
  },
  "强调色配置": {...}
}
```

**主入口获取主题**：
```python
# 主入口.py - 正确做法
def _get_theme_colors(self) -> Dict[str, str]:
    return self._config_service.get_theme_colors()  # ✓ 从配置服务获取
```

**基础模块定义fallback**：
```python
# 下拉框.py - 正确做法
DEFAULT_THEME = {
    "text_primary": "#000000",
    ...
}

def create(self, ..., theme_colors: Dict = None):
    if theme_colors is None:
        theme_colors = DEFAULT_THEME  # ✓ 使用fallback
```

### 7.4 错误示例

**错误：主入口定义主题配置**：
```python
# 主入口.py - 错误做法
def _get_theme_colors(self):
    themes = {
        "light": {...},  # ✗ 主入口定义主题配置
        "dark": {...},
    }
```

**错误：功能界面硬编码主题**：
```python
# 功能界面.py - 错误做法
def create(..., theme_colors: Dict = None):
    if theme_colors is None:
        theme_colors = {"text_primary": "#000000", ...}  # ✗ 硬编码
```

### 8.3 典型失误案例（2026-03-30）

#### 案例1：集资界面功能缺失

**问题描述**：
- V1版本：2个卡片（小号上贡、分城纳租），6个控件
- V2版本：1个卡片，2个控件
- **缺失**：分城纳租卡片、上贡限级、上贡限量、分城等级、纳租限量

**根本原因**：
1. 重构时未对比V1版本代码
2. 凭记忆重构，遗漏功能
3. 提交前未自主验证

**正确做法**：
```python
# 重构前必须读取V1版本
# V1路径：前端/游戏设置界面/表示层/界面/集资配置区.py
# 记录V1卡片配置：
# - 小号上贡：上贡限级、上贡限量、主要统帅、次要统帅
# - 分城纳租：分城等级、纳租限量
```

#### 案例2：打扫界面卡片缺失

**问题描述**：
- V1版本：2个卡片（打扫城区、打扫政区）
- V2版本：1个卡片（打扫配置）
- **缺失**：打扫政区卡片

#### 案例3：打野界面控件缺失

**问题描述**：
- V1版本：2个控件（野怪等级、搜索频率）
- V2版本：1个控件（打野模式）
- **缺失**：野怪等级、搜索频率控件

#### 案例4：冗余文件未清理

**问题描述**：
- 存在`其他界面.py`，包含重复的功能界面类
- 各功能界面已有独立模块文件

**正确做法**：
- 开发完成后检查是否有重复文件
- 及时删除不再使用的文件
