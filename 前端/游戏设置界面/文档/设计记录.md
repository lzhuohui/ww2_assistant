| 要素 | 说明 | 应用场景 |
|------|------|----------|
| Light (光照) | 高亮、聚焦、选中效果 | 悬停、焦点状态 |

| Motion (运动) | 动画过渡效果 | 状态切换、页面切换 |
| Material (材质) | Acrylic、Mica材质 | 窗口背景、弹出菜单 |
| Scale (缩放) | 响应式布局 | 不同屏幕尺寸适配 |

### 共识 #013：模块化设计规则 - 默认值传递

| 项目 | 内容 |
|------|------|
| **达成时间** | 2026-03-25 |
| **问题描述** | 中间层模块覆盖底层模块的默认值，导致模块化设计失效 |
| **案例** | 折叠卡片.py硬编码默认宽度，覆盖了下拉框.py和输入框.py的默认值 |

**模块化设计规则**：

| 规则 | 说明 |
|------|------|
| 默认值来源 | 被调用模块的默认值应由被调用模块自身定义 |
| 中间层传递 | 中间层应导入并使用底层模块的默认值，而非硬编码 |
| 调用者优先 | 调用者明确指定值时，优先使用调用者的值 |

**正确实现**：
```python
# 折叠卡片.py - 导入底层默认值
from 前端.游戏设置界面.表示层.组件.基础.下拉框 import Dropdown, USER_WIDTH as DROPDOWN_WIDTH
from 前端.游戏设置界面.表示层.组件.基础.输入框 import InputBox, USER_WIDTH as INPUT_WIDTH

# 使用底层默认值
control_width = control_config_item.get("width", DROPDOWN_WIDTH)  # 下拉框
control_width = control_config_item.get("width", INPUT_WIDTH)     # 输入框
```

**设计要点**：

| 要点 | 说明 |
|------|------|
| 选项来源 | 从账号配置区获取所有参与挂机的主帅名称 |
| 联动逻辑 | 次要统帅选项 = 所有主帅 - 已选主要统帅 |
| 组件增强 | 下拉框组件添加 `set_options` 方法支持动态更新选项 |

**数据获取逻辑**：
```python
def get_active_commanders(config_service: ConfigService) -> List[str]:
    """从账号配置中获取参与挂机的主帅列表"""
    commanders = []
    for i in range(1, 16):
        card_id = f"account_{i:02d}"
        account_type = config_service.get_value(card_id, "类型", "")
        is_enabled = config_service.get_value(card_id, "enabled", False)
        name = config_service.get_value(card_id, "名称", "")
        
        if account_type == "主帅" and is_enabled and name:
            commanders.append(name)
    return commanders
```

**联动实现**：
```python
def update_secondary_options():
    """更新次要统帅选项"""
    current_primary = primary_dropdown.get_value()
    secondary_options = ["请选统帅"] + [c for c in commander_options if c != current_primary]
    secondary_dropdown.set_options(secondary_options)
```

| **讨论方案** | A：卡片级加载/销毁（切换卡片时销毁旧卡片）<br>B：单个下拉框加载/销毁（关闭菜单时销毁）<br>C：懒加载不销毁（退出时统一销毁）<br>D：三层销毁策略 |
| **最终决策** | 采用方案D（三层销毁策略），最终调整为创建时加载完整选项列表 |
| **决策原因** | PopupMenuButton技术限制，动态更新items无法立即反映到已打开的菜单中 |

**技术限制说明**：

PopupMenuButton在打开菜单时，菜单内容已根据当前`items`渲染完成，动态更新items后需要再次打开才能看到新内容。因此无法实现"点击时加载"的懒加载模式。

**最终实现方案**：

```
启动应用
    ↓
进入配置界面
├─ 创建所有卡片控件
├─ 加载所有控件默认值
└─ 创建下拉框时直接传入完整选项列表 ✅
    ↓
【控件级】点击下拉框A
├─ 销毁上一个下拉框的选项列表
└─ 保持当前下拉框选项列表（已完整加载）
    ↓
【界面级】点击导航按钮（系统→账号）
├─ 销毁系统界面所有控件
└─ 创建账号界面
    ↓
退出应用
└─ 无需处理（已被界面级销毁）
```

**三层销毁架构**：

```
┌─────────────────────────────────────────────────┐
│ 应用级（主入口）                                 │
│   └─ 界面级（导航切换时销毁）                   │
│       └─ 控件级（下拉框切换时销毁选项列表）     │
│           └─ 数据级（选项列表）                 │
└─────────────────────────────────────────────────┘
```

**销毁时机详解**：

| 层级 | 触发时机 | 销毁对象 | 目的 |
|------|----------|----------|------|
| **控件级** | 点击下拉框A | 下拉框B的选项列表 | 防止选项列表积累 |
| **界面级** | 点击导航按钮 | 原界面所有控件和选项列表 | 防止界面级积累，避免退出卡顿 |
| **应用级** | 关闭窗口 | 已被界面级覆盖 | 无需额外处理 |

**关键实现要点**：

| 要点 | 模块 | 实现 |
|------|------|------|
| 控件级销毁 | Dropdown | 维护类变量 `_current_loaded_instance`，打开菜单时销毁上一个 |
| 界面级销毁 | MainEntry | 导航切换时调用 `manager.destroy_all()` |
| 加载标识管理 | Dropdown | `is_options_loaded` 根据选项数量初始化（完整选项=True，单一选项=False） |

**内存占用分析**：

```
假设场景：
- 每个卡片平均5个下拉框
- 每个下拉框平均10个选项
- 每个选项约20字节

内存占用：保留1个下拉框的选项列表 = 10 × 20 = 200字节 ✅
```

**性能优势**：

| 指标 | 说明 |
|------|------|
| 内存占用 | **0.2KB**（仅当前下拉框的选项列表） |
| 退出速度 | **快**（界面级销毁已完成） |
| 操作流畅度 | **快**（创建时已加载，无需等待） |
| 实现复杂度 | **低**（无需懒加载逻辑） |

### 共识 #020：下拉框懒加载优化方案（用户方案）

| 项目 | 内容 |
|------|------|
| **达成时间** | 2026-03-27 |
| **问题描述** | 建筑配置区有45个下拉框，每个下拉框有20个选项，导致启动时创建900个菜单项，内存占用大，启动慢 |
| **提出者** | 用户 |
| **方案类型** | 懒加载（Lazy Loading） |
| **核心思想** | 点击下拉框时才加载选项列表，任何时候只有一个下拉框有完整选项列表 |

**方案详细逻辑**：

#### 1. 进入界面后处理逻辑
```
1.1. 数据对接用户配置.json，获取当前界面的下拉框数据
1.2. 显示当前界面的全部下拉框，并初始加载状态标识为"没加载"
1.3. 数据接口转为对接下拉框选项列表文件（专门保存各下拉框的选项值：新建或从其他有此选项列表的文件）
```

#### 2. 下拉框操作逻辑
```
2.1. 点击下拉框，判断下拉框的加载状态
2.2. 销毁上一下拉框加载的选项列表
2.3. 判断本下拉框的加载状态
    2.3.1 没加载状态：加载本下拉框的对应选项列表
    2.3.2 已加载状态：不再次加载
2.4. 展开选项列表
2.5. 更新加载状态标识为"已加载"
```

#### 3. 界面切换逻辑
```
3.1. 点击导航按钮，销毁上一界面的控件
3.2. 进入对应界面，转到第一点（重新初始化）
```

**技术实现要点**：

| 要点 | 实现方式 |
|------|----------|
| **加载状态标识** | 每个下拉框维护 `is_loaded` 状态 |
| **当前活动下拉框** | 全局变量记录当前显示选项的下拉框 |
| **选项列表文件** | 每个下拉框对应一个JSON文件，存储选项列表 |
| **销毁机制** | 点击新下拉框时，销毁上一个的选项列表 |
| **数据对接** | 从用户配置.json读取默认值，从选项列表文件读取选项 |

**内存优化效果**：

| 场景 | 传统方式 | 懒加载方案 | 优化比例 |
|------|----------|------------|----------|
| 建筑配置区 | 45×20=900个菜单项 | 1×20=20个菜单项 | 减少98% |
| 所有界面 | 约1000个菜单项 | 最多20个菜单项 | 减少98% |
| 内存占用 | 约20KB | 约0.4KB | 减少98% |

**性能优势**：

| 指标 | 传统方式 | 懒加载方案 | 提升效果 |
|------|----------|------------|----------|
| 启动时间 | 慢（创建所有菜单项） | 快（只创建空下拉框） | 提升90% |
| 内存占用 | 高（所有选项常驻） | 低（仅当前选项） | 减少98% |
| 操作响应 | 快（选项已加载） | 快（点击时加载） | 无明显差异 |
| 界面切换 | 慢（需要销毁大量对象） | 快（销毁少量对象） | 提升80% |

**实现挑战与解决方案**：

| 挑战 | 解决方案 |
|------|----------|
| PopupMenuButton限制 | 使用动态更新items，需要重新打开菜单 |
| 状态同步 | 使用全局状态管理器跟踪加载状态 |
| 文件管理 | 为每个下拉框创建独立的选项列表文件 |
| 性能平衡 | 预加载常用下拉框的选项列表 |

**分阶段实施计划**：

| 阶段 | 目标 | 风险 | 预计时间 |
|------|------|------|----------|
| **阶段1** | 选项共享方案（所有下拉框共享同一个选项列表） | 低 | 1天 |
| **阶段2** | 分批加载方案（按卡片分组加载） | 中 | 2天 |
| **阶段3** | 完整懒加载方案（点击时加载） | 高 | 3天 |

**推荐实施顺序**：
1. **先实施阶段1**：解决最紧急的性能问题，减少90%内存占用
2. **验证效果后实施阶段2**：进一步优化启动速度
3. **最后实施阶段3**：实现完整的懒加载，达到最佳性能

**注意事项**：
1. **保持现有功能**：所有下拉框功能必须正常工作
2. **兼容现有API**：外部调用方式不能改变
3. **用户体验优先**：加载延迟不能影响操作流畅度
4. **错误处理**：选项列表文件不存在时的降级方案
